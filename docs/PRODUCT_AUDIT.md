# Super Dev Product Audit

更新时间：2026-03-10

## 结论

Super Dev 现在已经具备可发布、可交付、可继续扩展的产品骨架。

强项已经明确：

1. 宿主接入、审计、修复、Smoke 验收已经形成闭环
2. 主流水线已经从“提示词约束”升级成执行层硬门禁
3. 文档、Spec、前端运行验证、UI Review、质量门禁、交付、发布演练已经串成完整链路
4. 本地知识库已经进入 research、三文档、Spec、实现、质量和交付阶段

当前真正还缺的，不是更多零散功能，而是把少数高价值缺口继续做硬。

## 现状判断

### 已经做对的部分

1. 产品定位清楚
- 宿主负责模型、联网、编码、执行
- Super Dev 负责流程治理、设计约束、质量门禁、审计和交付标准

2. 首次开发路径完整
- research
- 三文档
- 用户确认
- Spec / tasks
- 前端优先与运行验证
- 后端 / 测试
- 质量门禁
- 交付与发布演练

3. 返工路径已经进入正式产品能力
- 文档确认门已经正式化
- UI 改版返工回路已经正式化
- 架构返工回路已经正式化
- 质量返工回路已经正式化

4. 宿主接入能力已经成体系
- `detect`
- `onboard`
- `doctor`
- `smoke`
- `audit`

### 当前最大的产品风险

1. 用户会把“接入成功”误解成“宿主已真实按完整流程工作”
2. 部分宿主问题并不是文件没生成，而是宿主真实会话行为还没有做完正式真人验收矩阵
3. 仓库边界如果不持续收紧，运行时宿主目录和源码会混在一起，后续维护成本会上升

## P0 缺口

### P0-1. 逐宿主真人会话验收矩阵还不完整

现在最强的是 `integrate audit`，它能校验：

- rules / commands / AGENTS / steering / workflows / skills 是否存在
- 内容是否带完整 Super Dev 契约
- 触发方式是否正确

这已经比“只看文件在不在”强很多。

但它仍然不是完整的真人会话验收。

仍需正式补齐的矩阵维度：

1. 宿主第一次收到 `/super-dev` 或 `super-dev:` 后，是否先进入 `research`
2. 是否真实写入：
- `output/*-research.md`
- `output/*-prd.md`
- `output/*-architecture.md`
- `output/*-uiux.md`
3. 是否在三文档后真实暂停等待确认
4. 确认后是否真实恢复到 Spec / 前端 / 后端阶段

为什么这是 P0：

- 这件事直接决定“除了 Claude Code 外，其它宿主是不是也真正可用”

### P0-2. 宿主前置条件诊断仍需更强

典型例子：

- iFlow API key 无效
- 宿主未登录
- 宿主旧会话没重载规则
- 宿主联网权限未开启

这些问题不一定是 Super Dev 核心逻辑 bug，但用户会把它理解成“接入失败”。

当前还需要更强的产品化处理：

1. 宿主前置条件提示
2. 宿主特定报错映射
3. `doctor` 更明确的行动建议

## P1 缺口

### P1-1. Proof Pack 已落地，仍需继续强化为对外信任资产

现在这些能力已经存在：

- docs confirmation
- frontend runtime
- UI review
- quality gate
- delivery ready
- release readiness

当前已经可通过 `super-dev release proof-pack` 聚合成正式交付证据包。

这一轮已经把 Proof Pack 接进了 Web 管理台主界面，能够直接看到：

- 交付证据完整度
- 当前阻塞项
- 关键交付证据
- 完整证据明细

下一阶段还需要继续强化为更强的对外信任资产：

- Proof of Research
- Proof of Spec
- Proof of Frontend Readiness
- Proof of Quality
- Proof of Delivery

这会比单独强调很多命令更容易建立信任。

### P1-2. 数据模型与安全返工仍未单独产品化

现在已经正式化的返工回路有：

1. 文档确认门
2. UI 改版回路
3. 架构返工回路
4. 质量返工回路

仍可继续增强的回路：

1. 数据模型返工
2. 更细粒度的安全整改回路

这两项当前仍可以通过质量返工和架构返工吸收，但还没有独立命令与状态文件。

### P1-3. 发布边界治理需要持续收紧

这一轮已经把常见运行时宿主目录从仓库边界里剥离到 `.gitignore`。

后续仍要坚持这条原则：

1. 版本化：
- 源码
- 测试
- 文档
- 正式变更 spec

2. 不版本化：
- 宿主运行时接入目录
- review-state
- output
- artifacts

如果这条边界松掉，后面会继续出现“本机接入产物混进仓库”的问题。

## P2 缺口

### P2-1. 设计导出能力仍是占位

当前代码里仍存在未实现导出：

1. Sketch export
2. Figma export

它们不会阻塞主流程，但如果对外继续保留这类能力名称，就容易造成预期偏差。

处理方式只能三选一：

1. 实现
2. 明确禁用
3. 降级说明

### P2-2. Spec 仍允许出现占位内容

当前 `specs/models.py` 在信息不足时会产出 `TBD`。

这对开发期容错有帮助，但对正式商业交付不够强。

建议后续增加：

1. Spec 占位符检查
2. 交付前禁止 `TBD`

目标：

1. 继续强化证据分类与摘要
2. 在 Web / CLI / 文档 / 官网中统一表达

### 阶段 3：继续细化返工回路

目标：

1. 数据返工
2. 更细粒度的安全整改回路
