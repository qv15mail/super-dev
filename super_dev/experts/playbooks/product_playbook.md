# 产品负责人（PRODUCT）操作手册

## 概述

产品负责人负责从用户视角审视产品的完整度、可用性和可信度。核心职责包括产品闭环审查、首次上手路径评估、功能缺口分析、断链流程识别、优先级决策、交付可信度评估。产品审查不是功能检查清单，而是站在真实用户的角度验证产品是否能够兑现其承诺的价值。

### 核心原则

1. **用户价值优先**：每个功能必须回答"为用户解决了什么问题"
2. **端到端闭环**：功能不只是实现了，还要可被发现、可被理解、可被使用、可被信赖
3. **最小可信交付**：MVP不是功能最少，而是信任链条最完整的最小集合
4. **数据驱动决策**：优先级和取舍基于用户数据和市场证据，而非直觉

## 方法论

### 一、产品闭环审查方法

#### 1.1 闭环六要素

```
对于每个核心功能，验证完整闭环:

发现 → 理解 → 操作 → 反馈 → 复用 → 信任

1. 发现（Discovery）:
   用户能否找到这个功能？
   入口是否明显？是否在用户预期的位置？
   新用户是否在首次使用中就能发现？

2. 理解（Comprehension）:
   用户是否理解这个功能做什么？
   界面文案是否清晰？是否用用户语言？
   是否需要额外解释（帮助文本/引导）？

3. 操作（Execution）:
   用户能否顺利完成操作？
   步骤数是否合理（核心操作<=5步）？
   是否有不必要的中断或跳转？

4. 反馈（Feedback）:
   操作后是否有明确反馈（成功/失败/进行中）？
   错误信息是否帮助用户修复问题？
   异步操作是否有状态通知？

5. 复用（Retention）:
   用户是否会再次使用？
   是否记住了用户偏好/历史？
   重复操作是否有快捷路径？

6. 信任（Trust）:
   用户是否信任操作结果？
   数据是否可验证？
   是否有撤销/回退机制？
```

#### 1.2 闭环评估矩阵

```
对每个核心功能打分（0-2分）:
  0 = 缺失或严重不足
  1 = 基本满足但有明显改进空间
  2 = 良好

功能闭环得分 = 各要素得分之和 / 12 x 100%

评判标准:
  >= 85%: 发布就绪
  70-84%: 需改进后发布
  < 70%: 不建议发布
```

### 二、首次上手路径评估

#### 2.1 FTUE（First Time User Experience）审查

```
审查流程:
  1. 以新用户身份清空所有缓存/状态
  2. 从落地页/首页开始
  3. 记录到达"首个价值时刻"(Aha Moment)的完整路径
  4. 评估每个步骤的必要性和摩擦度

评估维度:
  步骤数: 注册到首个价值时刻 <= 5步
  时间: 整个路径 <= 3分钟
  决策点: 需要用户做选择的地方 <= 2个
  输入量: 必填字段最小化
  阻断点: 是否有任何步骤导致用户无法继续
```

#### 2.2 价值时刻定义

```
不同产品的价值时刻:
  电商: 找到想要的商品并加入购物车
  SaaS工具: 用工具完成一个真实的任务
  社交产品: 关注/互动了一个有趣的人
  内容平台: 阅读/观看了一个有价值的内容

评估标准:
  价值时刻是否在注册后立即可达？
  空状态是否引导用户走向价值时刻？
  是否有示例数据/模板帮助用户快速体验价值？
```

#### 2.3 流失漏斗分析

```
漏斗阶段:
  访问 → 注册开始 → 注册完成 → 首次核心操作 → 首次成功 → 7日留存

每个阶段检查:
  转化率是否合理（基准对比）
  流失用户在哪个步骤放弃
  放弃原因是什么（技术障碍/理解困难/缺乏动力）
  是否有挽回机制（邮件提醒/引导优化）
```

### 三、功能缺口分析

#### 3.1 功能矩阵

```
对标维度:
  核心功能: 用户选择这个产品必须有的功能
  差异功能: 与竞品形成差异化的功能
  卫生功能: 用户默认期待存在的基础功能

评估方法:
  1. 列出目标用户的核心需求清单
  2. 列出主要竞品的功能矩阵
  3. 对比自身产品的覆盖度
  4. 标识缺口（Must-have缺失 = 阻断性缺口）

缺口分类:
  阻断性缺口: 缺少此功能用户无法使用产品 → 必须在MVP前补齐
  体验性缺口: 有替代方案但体验差 → 计入迭代计划
  期望性缺口: 用户期望有但不影响核心流程 → 纳入路线图
```

#### 3.2 功能完整度评估

```
每个功能检查:
  正常路径: 是否可以完成核心操作
  异常路径: 错误/异常是否有合理处理
  边界场景: 大数据量/特殊字符/并发是否正常
  权限控制: 不同角色是否有正确的访问控制
  数据一致性: 操作结果是否持久化且可验证
  可撤销性: 是否可以回退操作（尤其是危险操作）
```

### 四、断链流程识别

#### 4.1 断链定义

```
断链: 用户在完成任务的路径上遭遇中断，无法继续前进或需要非预期的操作才能继续。

常见断链类型:
  死胡同: 页面没有下一步引导，用户不知道做什么
  断裂跳转: 操作完成后跳转到不相关的页面
  状态丢失: 填写表单中途刷新/返回导致数据丢失
  权限墙: 操作到一半发现需要额外权限/付费
  外部依赖: 需要离开产品完成某步骤（如到邮箱验证）
  功能缺失: 按钮/链接存在但功能未实现
  循环引用: A页面引导去B，B又引导回A
```

#### 4.2 断链检测方法

```
路径走查法:
  1. 列出所有用户核心路径（如注册→创建→编辑→发布→分享）
  2. 逐步模拟操作，记录每个页面的:
     - 用户可执行的操作
     - 每个操作后的页面状态
     - 是否有明确的下一步引导
  3. 标识所有中断点

端到端验证:
  正向走查: 从入口到完成的完整路径
  逆向走查: 从完成状态回退到起点
  中途退出: 在任意步骤中断后重新进入
  异常走查: 在每个步骤故意制造错误
```

### 五、优先级决策框架

#### 5.1 RICE评分模型

```
RICE = (Reach x Impact x Confidence) / Effort

Reach（覆盖面）: 受影响用户数/时间段
  示例: 每月500个用户会触发 → Reach = 500

Impact（影响度）: 对单个用户的影响程度
  3 = 巨大影响（解决核心痛点）
  2 = 较大影响（明显改善体验）
  1 = 中等影响（有帮助但非必需）
  0.5 = 轻微影响
  0.25 = 微小影响

Confidence（置信度）: 对预估的信心
  100% = 有数据支撑
  80%  = 有用户反馈
  50%  = 直觉判断

Effort（工作量）: 人月为单位
  示例: 2个工程师1周 = 0.5人月

RICE分值越高，优先级越高
```

#### 5.2 优先级矩阵

```
紧急 + 重要: 立即执行（P0缺陷/安全漏洞/核心流程断链）
不紧急 + 重要: 计入当前迭代（体验改进/功能补齐）
紧急 + 不重要: 评估是否可以简化处理
不紧急 + 不重要: 记入Backlog，定期清理

快赢原则:
  Effort <= 0.5人天 + Impact >= 2 → 立即做（快赢）
  无论Backlog优先级如何，快赢项优先执行
```

### 六、交付可信度评估

#### 6.1 可信度维度

```
技术可信度:
  - 核心功能是否全部可用（非mock/stub）
  - 性能是否达到承诺指标
  - 安全是否通过审查
  - 监控和告警是否就位

产品可信度:
  - 功能是否端到端闭环
  - 用户路径是否无断链
  - 空状态/错误状态是否有设计
  - 文案和措辞是否专业

运营可信度:
  - 帮助文档/FAQ是否就绪
  - 客服/反馈渠道是否建立
  - 数据统计和分析是否可用
  - 应急预案是否准备
```

#### 6.2 发布就绪评估

```
Must-have（阻断发布）:
  - 核心用户路径全部可用
  - P0/P1缺陷全部修复
  - 安全审查通过
  - 性能达到基线
  - 回滚方案验证

Should-have（影响发布质量）:
  - 所有状态设计实现（空/加载/错误/成功）
  - 帮助文档就绪
  - 监控仪表盘就位
  - 用户反馈渠道建立

Nice-to-have（后续迭代）:
  - 高级功能/个性化
  - 性能优化（超出基线的提升）
  - A/B测试配置
```

## 产品审查检查清单

### 闭环审查

- [ ] 每个核心功能是否可被发现（入口明显/位置合理）
- [ ] 功能描述是否使用用户语言（非技术术语）
- [ ] 核心操作是否 <= 5步完成
- [ ] 操作后是否有明确反馈（成功/失败/进行中）
- [ ] 异步操作是否有状态通知
- [ ] 重要操作是否可撤销
- [ ] 数据是否可验证和导出

### 首次上手审查

- [ ] 注册流程是否最小化（<=3步/<=5个必填字段）
- [ ] 注册后是否立即引导到价值时刻
- [ ] 空状态是否有引导和示例数据
- [ ] 是否提供跳过/稍后选项
- [ ] 是否有渐进式引导（不一次性展示所有功能）

### 断链审查

- [ ] 每个页面是否有明确的下一步行动
- [ ] 表单中途退出后是否保留数据
- [ ] 操作完成后是否跳转到合理位置
- [ ] 权限不足是否提前提示（而非操作后拒绝）
- [ ] 所有按钮/链接是否都有功能（无dead link）
- [ ] 是否有循环引用的页面

### 功能完整度审查

- [ ] 核心功能是否全部可用（非mock/stub）
- [ ] 异常路径是否有合理处理
- [ ] 大数据量下是否有分页/加载更多
- [ ] 搜索/筛选是否返回合理结果
- [ ] 不同角色/权限是否正确控制
- [ ] 移动端是否可用

### 可信度审查

- [ ] 产品是否看起来"完整"（非半成品感）
- [ ] 文案/措辞是否一致且专业
- [ ] 有无拼写错误/翻译遗漏
- [ ] 价格/费用是否明确透明
- [ ] 是否有社会证明（案例/评价/数据）
- [ ] 隐私/条款是否可访问

## 交叉审查要点

### 与UX设计师交叉

- 用户旅程是否覆盖了完整闭环
- 首次上手路径是否经过UX评审
- 状态设计是否满足产品期望

### 与UI设计师交叉

- 视觉表现是否传达了产品定位
- CTA按钮是否足够突出
- 信任模块（证书/案例/数据）是否在合适位置

### 与QA交叉

- 所有核心路径是否有E2E测试覆盖
- 断链问题是否纳入回归测试
- 边界场景是否测试过

### 与架构师交叉

- 架构是否支持产品的扩展计划
- 性能是否满足产品承诺的体验
- 数据模型是否支持未来的功能需求

## 常见反模式

| 反模式 | 问题 | 正确做法 |
|-------|------|---------|
| 功能堆砌 | 功能多但每个都半成品 | 少而精，每个功能闭环完整 |
| 开发者视角 | 用系统术语而非用户语言 | 所有文案使用用户的语言 |
| 价值延迟 | 注册后看到空白页面 | 注册后立即引导到价值时刻 |
| 断链放任 | 按钮存在但跳转到404 | 未完成功能不暴露入口 |
| 指标虚荣 | 只看注册数不看留存 | 关注端到端转化漏斗 |
| 闭门造车 | 不看竞品不问用户 | 定期用户访谈和竞品分析 |
| 发布焦虑 | 带着已知严重问题发布 | Must-have不达标就不发 |
| 忽视信任 | 产品看起来像半成品 | 文案/设计/完整度建立信任 |

## 知识依据

- **闭环审查**：Super Dev产品审计方法论、双钻石设计模型
- **FTUE**：Nir Eyal《Hooked》、Samuel Hulick《The Elements of User Onboarding》
- **优先级**：RICE框架（Intercom）、MoSCoW方法、Kano模型
- **断链分析**：认知走查法（Cognitive Walkthrough）、启发式评估
- **可信度**：BJ Fogg行为模型、Nielsen Norman Group信任研究
- **功能缺口**：JTBD（Jobs to Be Done）、Strategyzer价值主张画布
- **度量指标**：AARRR海盗指标、North Star Metric框架
