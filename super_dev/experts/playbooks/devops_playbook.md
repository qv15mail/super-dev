# DevOps专家（DEVOPS）操作手册

## 概述

DevOps专家负责构建和维护高效、可靠的软件交付流水线。核心职责包括CI/CD流水线设计、容器化标准制定、部署策略规划、监控告警体系建设、基础设施即代码管理。DevOps不是一个岗位，而是一种打通开发与运维壁垒的文化和实践。

### 核心原则

1. **自动化一切**：可重复的操作必须自动化，消除人工干预风险
2. **基础设施即代码**：所有基础设施配置必须版本化、可审计、可复现
3. **快速反馈**：从代码提交到生产部署的每个环节提供即时反馈
4. **不可变基础设施**：部署的产物不可修改，变更通过替换而非修补实现

## 方法论

### 一、CI/CD流水线设计

#### 1.1 标准流水线阶段

```
代码提交 → 构建 → 静态分析 → 单元测试 → 构建镜像 → 安全扫描
    → 集成测试 → 预发布部署 → E2E测试 → 灰度发布 → 全量发布 → 发布后验证
```

#### 1.2 各阶段详细规范

| 阶段 | 触发条件 | 超时 | 失败处理 | 产出物 |
|------|---------|------|---------|--------|
| 构建 | 代码推送/PR创建 | 5min | 阻断，通知开发者 | 构建产物 |
| 静态分析 | 构建成功 | 3min | 阻断（error级别） | lint报告 |
| 单元测试 | 构建成功 | 10min | 阻断 | 测试报告+覆盖率 |
| 构建镜像 | 测试通过 | 10min | 阻断 | Docker镜像+tag |
| 安全扫描 | 镜像构建完成 | 5min | 高危阻断，中低警告 | 漏洞报告 |
| 集成测试 | 安全扫描通过 | 15min | 阻断 | 集成测试报告 |
| 预发布部署 | 集成测试通过 | 5min | 阻断，自动回滚 | 部署状态 |
| E2E测试 | 预发布部署成功 | 20min | 阻断 | E2E测试报告 |
| 灰度发布 | E2E通过+人工审批 | 30min | 自动回滚 | 灰度指标 |
| 全量发布 | 灰度验证通过 | 10min | 自动回滚 | 发布记录 |
| 发布后验证 | 全量发布完成 | 15min | 告警+准备回滚 | 健康检查报告 |

#### 1.3 分支策略与流水线映射

```
main分支:
  触发: 全量流水线（含灰度和生产部署）
  保护: 需要PR审批 + 所有检查通过

develop分支:
  触发: 构建 → 测试 → 安全扫描 → 部署到开发环境
  保护: 需要PR审批

feature/*分支:
  触发: 构建 → 测试 → 静态分析
  保护: 无

hotfix/*分支:
  触发: 全量流水线（加速通道，跳过灰度等待期）
  保护: 需要高级审批
```

#### 1.4 流水线优化

```
并行化:
  - 静态分析与单元测试并行执行
  - 多平台构建并行（amd64/arm64）
  - 测试分片（test sharding）并行执行

缓存策略:
  - 依赖缓存（npm cache / pip cache / go mod cache）
  - Docker层缓存（BuildKit cache mount）
  - 构建产物缓存（增量构建）

加速手段:
  - 变更检测：只构建受影响的模块（monorepo）
  - 测试影响分析：只运行受变更影响的测试
  - 预构建基础镜像：减少构建时间
```

### 二、容器化标准

#### 2.1 Dockerfile 最佳实践

```dockerfile
# 多阶段构建示例
# 阶段1：构建
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --production=false
COPY . .
RUN npm run build

# 阶段2：运行
FROM node:20-alpine AS runner
RUN addgroup -g 1001 -S appgroup && \
    adduser -S appuser -u 1001 -G appgroup
WORKDIR /app
COPY --from=builder --chown=appuser:appgroup /app/dist ./dist
COPY --from=builder --chown=appuser:appgroup /app/node_modules ./node_modules
COPY --from=builder --chown=appuser:appgroup /app/package.json ./

USER appuser
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

CMD ["node", "dist/main.js"]
```

#### 2.2 镜像安全规范

```
基础镜像选择:
  - 优先使用 Alpine 变体（体积小、攻击面小）
  - 固定基础镜像版本（禁止使用 latest 标签）
  - 定期更新基础镜像（每月安全补丁）

安全扫描:
  - 构建时扫描: Trivy / Grype
  - 高危漏洞（CRITICAL/HIGH）阻断构建
  - 中低漏洞限期修复（7天/30天）

最小权限:
  - 使用非root用户运行应用
  - 只复制运行所需的文件（不含源代码/测试/文档）
  - 不安装不必要的包（curl/wget按需添加仅用于健康检查）

秘密管理:
  - 禁止将密钥/证书硬编码到镜像中
  - 使用运行时注入（环境变量/Kubernetes Secret/Vault）
  - 构建参数（ARG）不用于传递密钥（会留在镜像层中）
```

#### 2.3 镜像标签策略

```
标签格式: {服务名}:{git-sha-short}-{构建号}
示例: api:a1b2c3d-42

额外标签:
  - latest: 仅用于开发环境
  - v{semver}: 正式版本标签
  - {分支名}-latest: 分支最新构建

禁止:
  - 生产环境使用 latest 标签
  - 覆盖已有的版本标签
  - 使用不可追溯的标签（如纯日期）
```

### 三、部署策略

#### 3.1 蓝绿部署

```
适用场景: 需要零停机切换的关键服务

流程:
  1. 蓝色环境运行当前版本（接收生产流量）
  2. 绿色环境部署新版本
  3. 对绿色环境运行健康检查和冒烟测试
  4. 切换负载均衡器指向绿色环境
  5. 监控绿色环境指标（5-15分钟）
  6. 确认无异常后销毁蓝色环境
  7. 异常时立即回切到蓝色环境

优点: 回滚速度极快（秒级）
缺点: 需要双倍资源
```

#### 3.2 灰度/金丝雀发布

```
适用场景: 大规模用户群的渐进式发布

流程:
  1. 部署新版本到灰度集群（不接收流量）
  2. 切入 5% 流量到新版本
  3. 观察关键指标（错误率/延迟/业务指标）持续10分钟
  4. 指标正常 → 扩大到 25% → 50% → 100%
  5. 任一阶段指标异常 → 立即回滚到 0%

自动化判定:
  错误率增长 > 0.5% → 自动回滚
  P99延迟增长 > 50% → 自动回滚
  自定义业务指标下降 > 阈值 → 告警并暂停
```

#### 3.3 滚动更新

```
适用场景: Kubernetes默认策略，适合无状态服务

配置:
  maxUnavailable: 25%   # 最多25%的Pod不可用
  maxSurge: 25%         # 最多额外创建25%的Pod
  minReadySeconds: 30   # Pod就绪后等待30秒再继续

健康检查:
  livenessProbe:  检测容器是否存活（失败则重启）
  readinessProbe: 检测容器是否可接收流量（失败则从Service摘除）
  startupProbe:   检测容器是否启动完成（启动慢的服务使用）
```

#### 3.4 回滚策略

```
自动回滚触发条件:
  - 健康检查连续失败 N 次
  - 错误率超过阈值
  - 部署超时（Pod无法就绪）

回滚流程:
  1. 停止当前发布
  2. 回退到上一个已知正常的版本
  3. 验证回滚后服务健康
  4. 通知相关团队
  5. 创建事件记录

回滚保障:
  - 数据库迁移必须向后兼容（不执行破坏性DDL）
  - API变更必须向后兼容（新旧版本共存期间）
  - 保留至少最近 5 个版本的部署产物
```

### 四、监控告警设计

#### 4.1 四黄金信号

```
延迟（Latency）:
  指标: 请求响应时间分布（P50/P95/P99）
  告警: P95 > 500ms 持续 5 分钟
  仪表盘: 时间序列图 + 分位线

流量（Traffic）:
  指标: QPS / RPM / 活跃连接数
  告警: 流量突降 > 50% 持续 2 分钟（可能是故障）
  仪表盘: 实时流量图 + 同比/环比

错误（Errors）:
  指标: HTTP 5xx 比率 / 业务错误率
  告警: 5xx 比率 > 1% 持续 1 分钟
  仪表盘: 错误率图 + 错误分类饼图

饱和度（Saturation）:
  指标: CPU/内存/磁盘/连接池使用率
  告警: CPU > 80% 持续 5 分钟 / 内存 > 85% / 磁盘 > 90%
  仪表盘: 资源使用率图 + 趋势预测
```

#### 4.2 SLO/SLI/SLA 设计

```
SLI（服务级别指标）:
  可用性SLI = 成功请求数 / 总请求数
  延迟SLI = P95延迟 < 500ms 的时间窗口比例

SLO（服务级别目标）:
  可用性SLO = 99.9%（月度）→ 允许 43.2 分钟停机
  延迟SLO = 99% 的请求 P95 < 500ms

SLA（服务级别协议）:
  SLO的外部承诺版本，通常比内部SLO宽松
  违反SLA可能涉及赔偿条款

错误预算:
  月度错误预算 = 1 - SLO = 0.1% = 43.2分钟
  预算消耗 > 50% → 减缓发布节奏
  预算消耗 > 80% → 冻结非关键发布
  预算耗尽 → 全力修复，停止新功能发布
```

#### 4.3 告警分级

| 等级 | 条件 | 响应方式 | 响应时间 |
|------|------|---------|---------|
| P0-紧急 | 服务完全不可用/数据丢失 | 电话+短信+即时通讯 | 5分钟内响应 |
| P1-严重 | 核心功能降级/性能严重下降 | 短信+即时通讯 | 15分钟内响应 |
| P2-警告 | 非核心功能异常/资源接近阈值 | 即时通讯 | 1小时内响应 |
| P3-提示 | 预警类信息/趋势异常 | 邮件/工单 | 下个工作日 |

#### 4.4 日志规范

```
日志级别使用:
  ERROR: 需要立即处理的错误（影响用户/数据）
  WARN:  潜在问题，暂不影响功能
  INFO:  关键业务操作（用户登录/订单创建/支付）
  DEBUG: 开发调试信息（生产环境默认关闭）

结构化日志格式（JSON）:
{
  "timestamp": "2024-01-15T10:30:00.123Z",
  "level": "ERROR",
  "service": "api-gateway",
  "trace_id": "abc123",
  "span_id": "def456",
  "message": "Failed to process payment",
  "error": "ConnectionTimeout",
  "user_id": "u_789",
  "duration_ms": 5000
}

日志禁忌:
  - 禁止记录密码、token、信用卡号等敏感信息
  - 禁止在循环中打印日志（日志洪水）
  - 禁止使用 fmt.Println / console.log 替代日志框架
  - 禁止日志消息中包含可变的高基数ID（导致日志聚合失效）
```

## DevOps检查清单

### CI/CD流水线审查

- [ ] 流水线是否覆盖完整阶段（构建→测试→扫描→部署→验证）
- [ ] 各阶段是否设置了合理的超时时间
- [ ] 失败是否有明确的通知机制
- [ ] 依赖和构建缓存是否配置
- [ ] 流水线执行时间是否在可接受范围内（<30分钟）
- [ ] 是否支持并行执行以加速流水线
- [ ] 分支保护规则是否配置正确

### 容器化审查

- [ ] Dockerfile是否使用多阶段构建
- [ ] 基础镜像是否固定版本（非latest）
- [ ] 是否使用非root用户运行
- [ ] 是否配置了健康检查（HEALTHCHECK）
- [ ] 镜像大小是否优化（<200MB为佳）
- [ ] 安全扫描是否集成到构建流程
- [ ] 是否遵循.dockerignore排除无关文件

### 部署策略审查

- [ ] 是否有明确的部署策略（蓝绿/灰度/滚动）
- [ ] 回滚方案是否验证过可行
- [ ] 数据库迁移是否向后兼容
- [ ] 部署过程是否零停机
- [ ] 灰度发布是否有自动判定和回滚机制
- [ ] 发布后是否有自动健康检查

### 监控告警审查

- [ ] 四黄金信号是否全部覆盖
- [ ] SLO/SLI是否明确定义
- [ ] 告警是否分级并配置了正确的通知渠道
- [ ] 是否有告警抑制和去重机制（避免告警风暴）
- [ ] 仪表盘是否覆盖关键业务指标
- [ ] 日志是否结构化并集中收集
- [ ] 分布式追踪是否接入

## 交叉审查要点

### 与架构师交叉

- 服务拆分粒度是否适合独立部署
- 服务间通信是否支持优雅降级
- 配置管理是否外部化（不硬编码到镜像）

### 与安全专家交叉

- 密钥管理是否使用Vault/KMS等专用工具
- 镜像扫描是否覆盖了已知CVE
- 网络策略是否限制了不必要的服务间通信

### 与QA交叉

- 测试环境是否与生产配置一致
- CI中测试阶段的资源分配是否充足
- 测试报告是否自动归档和展示

### 与开发交叉

- 本地开发环境是否可以一键启动（docker-compose）
- 环境变量/配置是否有文档说明
- 日志和监控SDK是否集成到应用中

## 常见反模式

| 反模式 | 问题 | 正确做法 |
|-------|------|---------|
| 雪花服务器 | 手动配置，不可复现 | 基础设施即代码（Terraform/Pulumi） |
| 部署恐惧 | 发布频率低，每次变更大 | 小批量频繁发布，自动化流水线 |
| 告警疲劳 | 告警过多导致忽略 | 精准告警，分级处理，定期优化 |
| 日志黑洞 | 日志无结构，无法检索 | 结构化日志+集中收集+索引 |
| 手动回滚 | 回滚靠人记忆和手工操作 | 自动化回滚脚本+一键回滚 |
| 配置漂移 | 各环境配置不一致 | GitOps，配置版本化管理 |
| 监控盲区 | 只监控基础设施，不监控业务 | 四黄金信号+业务指标全覆盖 |
| 大镜像 | 镜像包含编译工具/源码/测试 | 多阶段构建，最小化运行镜像 |
