# QA专家（QA）操作手册

## 概述

QA专家负责在软件开发全生命周期中保障产品质量。核心职责包括测试策略设计、测试用例编写、质量门禁管理、自动化测试推进、缺陷跟踪闭环。质量不是测试出来的，而是构建过程中每个环节共同保障的结果。

### 核心原则

1. **预防优于检测**：在缺陷产生之前通过流程和规范避免，而非事后修补
2. **风险驱动测试**：优先覆盖高风险、高影响的功能路径
3. **持续反馈**：测试结果快速反馈给开发团队，缩短修复周期
4. **可度量质量**：所有质量目标必须可量化、可追踪、可验证

## 方法论

### 一、测试策略设计

#### 1.1 测试金字塔

测试金字塔是分层测试策略的基础模型，从底层到顶层执行速度递减、维护成本递增：

| 层级 | 占比 | 执行速度 | 维护成本 | 覆盖目标 |
|------|------|---------|---------|---------|
| 单元测试 | 60-70% | <100ms/个 | 低 | 函数/方法逻辑正确性 |
| 集成测试 | 20-25% | <5s/个 | 中 | 模块间交互、API契约 |
| E2E测试 | 5-10% | <30s/个 | 高 | 关键用户路径、业务流程 |
| 手动探索测试 | 按需 | 不定 | 高 | 边界场景、用户体验 |

#### 1.2 风险矩阵评估

对每个功能模块进行风险评估，确定测试优先级：

```
风险等级 = 影响程度 × 发生概率

影响程度（1-5）:
  5 - 数据丢失/资金损失/安全漏洞
  4 - 核心功能不可用
  3 - 功能部分异常，有替代方案
  2 - 非核心功能异常
  1 - 视觉/文案等轻微问题

发生概率（1-5）:
  5 - 每次操作必现
  4 - 高频场景下大概率发生
  3 - 特定条件下发生
  2 - 罕见场景下发生
  1 - 极端边界条件
```

| 风险等级 | 分值范围 | 测试要求 |
|---------|---------|---------|
| 极高 | 20-25 | 单元+集成+E2E+回归+性能测试全覆盖 |
| 高 | 12-19 | 单元+集成+E2E覆盖 |
| 中 | 6-11 | 单元+集成覆盖 |
| 低 | 1-5 | 单元测试覆盖 |

#### 1.3 测试范围确定

```
输入：需求文档（PRD）、架构文档、UIUX设计稿
输出：测试范围矩阵

步骤：
1. 拆解功能点 → 列出所有可测试单元
2. 对每个功能点进行风险评估
3. 确定每个功能点的测试层级和覆盖方式
4. 标注测试依赖（数据、环境、第三方服务）
5. 估算测试工作量并排入计划
```

### 二、测试用例设计方法

#### 2.1 等价类划分

将输入域划分为若干等价类，每类取一个代表值进行测试：

```
示例：用户年龄输入（有效范围 18-120）

有效等价类：
  - 18-120 之间的整数 → 取值 25

无效等价类：
  - 小于 18 → 取值 15
  - 大于 120 → 取值 150
  - 非整数 → 取值 25.5
  - 非数字 → 取值 "abc"
  - 空值 → 取值 null
  - 负数 → 取值 -1
```

#### 2.2 边界值分析

在等价类的边界处取值测试，通常为边界值及其相邻值：

```
示例：用户年龄输入（有效范围 18-120）

边界值：
  - 17（刚好低于下限）→ 预期拒绝
  - 18（下限）→ 预期接受
  - 19（刚好高于下限）→ 预期接受
  - 119（刚好低于上限）→ 预期接受
  - 120（上限）→ 预期接受
  - 121（刚好高于上限）→ 预期拒绝
  - 0（零值）→ 预期拒绝
  - MAX_INT（极大值）→ 预期拒绝
```

#### 2.3 状态转换测试

对有状态的系统，测试所有合法和非法的状态转换：

```
示例：订单状态机

状态：待支付 → 已支付 → 备货中 → 已发货 → 已签收 → 已完成
                                                    ↘ 退货中 → 已退款

测试覆盖：
  正向转换：每条合法路径至少一个用例
  逆向转换：尝试回退到前一状态（预期拒绝）
  跳跃转换：尝试跳过中间状态（预期拒绝）
  并发转换：两个请求同时修改同一订单状态
  超时转换：支付超时自动取消
```

#### 2.4 判定表法

适用于多条件组合的场景，穷举所有条件组合及对应结果：

```
示例：折扣计算规则

条件：
  C1: 是否会员（Y/N）
  C2: 订单金额 ≥ 500（Y/N）
  C3: 是否首单（Y/N）

| 规则 | C1 | C2 | C3 | 折扣 |
|------|----|----|----| ------|
| R1   | Y  | Y  | Y  | 30%  |
| R2   | Y  | Y  | N  | 20%  |
| R3   | Y  | N  | Y  | 15%  |
| R4   | Y  | N  | N  | 10%  |
| R5   | N  | Y  | Y  | 15%  |
| R6   | N  | Y  | N  | 5%   |
| R7   | N  | N  | Y  | 10%  |
| R8   | N  | N  | N  | 0%   |
```

#### 2.5 测试用例模板

```yaml
用例编号: TC-{模块}-{序号}
优先级: P0/P1/P2/P3
前置条件:
  - 条件1
  - 条件2
测试步骤:
  1. 操作步骤1
  2. 操作步骤2
测试数据:
  字段1: 值1
  字段2: 值2
预期结果:
  - 预期1
  - 预期2
实际结果: (执行时填写)
通过状态: PASS/FAIL/BLOCKED
```

### 三、质量门禁标准

#### 3.1 各阶段质量门禁

| 阶段 | 通过条件 | 阻断条件 |
|------|---------|---------|
| 代码提交 | 单元测试通过率100%，lint无error | 任意单元测试失败 |
| PR合并 | 代码覆盖率≥80%，无P0/P1缺陷 | 覆盖率低于阈值 |
| 集成测试 | 所有API契约测试通过 | 契约不匹配 |
| 预发布 | E2E核心路径100%通过，性能基线达标 | 核心路径失败 |
| 生产发布 | 灰度验证通过，监控无异常 | 错误率上升 |
| 发布后 | 24h内无P0回归，用户反馈无严重问题 | P0缺陷出现 |

#### 3.2 覆盖率要求

```
整体代码覆盖率：≥ 80%
核心业务逻辑覆盖率：≥ 90%
新增代码覆盖率：≥ 85%
分支覆盖率：≥ 75%
API端点覆盖率：100%（每个端点至少正常+异常各一个用例）
```

#### 3.3 性能基线

```
API 响应时间：
  P50 < 200ms
  P95 < 500ms
  P99 < 1000ms

页面加载：
  FCP < 1.5s
  LCP < 2.5s
  TTI < 3.5s
  CLS < 0.1

并发能力：
  目标QPS不低于预估峰值的 2 倍
  错误率 < 0.1%
```

### 四、自动化测试策略

#### 4.1 自动化适用性判断

| 适合自动化 | 不适合自动化 |
|-----------|------------|
| 回归测试（重复执行） | 探索性测试 |
| 数据驱动测试 | 一次性验证 |
| API契约测试 | 视觉美观评估 |
| 性能/负载测试 | 用户体验主观评价 |
| 跨浏览器兼容性 | 需求尚未稳定的功能 |

#### 4.2 自动化测试分层

```
单元测试框架：
  Python: pytest + pytest-cov
  JavaScript: Jest / Vitest
  Go: testing + testify

集成测试框架：
  API测试: supertest / httpx / REST-assured
  数据库: testcontainers
  消息队列: embedded broker

E2E测试框架：
  Web: Playwright（首选）/ Cypress
  Mobile: Detox / Appium

性能测试：
  负载: k6 / Locust
  基准: pytest-benchmark / wrk
```

#### 4.3 测试数据管理

```
原则：
  1. 测试数据与生产数据完全隔离
  2. 每个测试用例独立创建所需数据（避免测试间依赖）
  3. 测试结束后清理（或使用事务回滚）
  4. 敏感数据使用脱敏/伪造数据（faker）
  5. 数据工厂模式统一管理测试数据生成

反模式：
  - 共享全局测试数据（测试顺序依赖）
  - 直接使用生产数据库快照（隐私风险）
  - 硬编码测试数据（维护成本高）
```

### 五、缺陷管理

#### 5.1 缺陷分级

| 等级 | 定义 | 修复SLA | 示例 |
|------|------|--------|------|
| P0-致命 | 系统不可用/数据丢失/安全漏洞 | 4小时内修复 | 生产宕机、SQL注入、支付错误 |
| P1-严重 | 核心功能不可用，无替代方案 | 1个工作日 | 登录失败、订单无法创建 |
| P2-一般 | 功能异常但有替代方案 | 3个工作日 | 导出格式错误、排序不正确 |
| P3-轻微 | 不影响功能的问题 | 下个迭代 | 文案错误、对齐偏差 |

#### 5.2 缺陷报告模板

```
标题: [P{等级}][模块] 简明描述
环境: 操作系统/浏览器/版本/分支
前置条件:
复现步骤:
  1. 步骤1
  2. 步骤2
预期结果:
实际结果:
截图/日志:
影响范围:
复现概率: 必现/高概率/低概率/偶现
```

#### 5.3 缺陷闭环流程

```
提交 → 确认 → 分配 → 修复中 → 修复完成 → 验证 → 关闭
  ↓                                        ↓
  驳回（非缺陷/重复）                      验证失败 → 重新打开

闭环要求：
  - 每个缺陷必须有明确的根因分析
  - 修复必须附带回归测试用例
  - P0/P1缺陷必须在修复后进行全量回归
  - 缺陷趋势每周统计并分析
```

#### 5.4 缺陷度量指标

```
缺陷密度 = 缺陷数 / KLOC（千行代码）
缺陷逃逸率 = 生产缺陷数 / 总缺陷数
修复时效 = 平均修复时长（按等级统计）
重开率 = 重新打开缺陷数 / 已关闭缺陷数
回归引入率 = 回归缺陷数 / 修复缺陷数
```

## QA审查检查清单

### 测试策略审查

- [ ] 是否为每个功能模块定义了测试层级（单元/集成/E2E）
- [ ] 测试金字塔比例是否合理（单元>集成>E2E）
- [ ] 是否完成风险矩阵评估并据此确定测试优先级
- [ ] 是否识别了所有关键业务路径并确保E2E覆盖
- [ ] 测试范围是否覆盖了PRD中所有功能需求
- [ ] 非功能需求（性能/安全/兼容性）是否有对应测试计划

### 测试用例审查

- [ ] 所有输入字段是否应用了等价类划分
- [ ] 所有数值/日期/字符串边界是否有边界值测试
- [ ] 有状态的业务流程是否覆盖了全部状态转换（含非法转换）
- [ ] 多条件组合场景是否使用判定表穷举
- [ ] 负面测试用例是否充分（非法输入/权限不足/网络异常）
- [ ] 并发场景是否有测试（竞态条件/死锁/数据一致性）
- [ ] 幂等性测试：重复提交是否安全
- [ ] 超时和重试场景是否覆盖

### 质量门禁审查

- [ ] 代码覆盖率是否达到阈值（整体≥80%，核心≥90%）
- [ ] 所有P0/P1缺陷是否在发布前修复并验证
- [ ] 性能基线是否满足（API P95<500ms，页面LCP<2.5s）
- [ ] 安全扫描是否通过（无高危漏洞）
- [ ] 回归测试是否全部通过
- [ ] 新增代码覆盖率是否达标（≥85%）

### 自动化测试审查

- [ ] 自动化测试是否可在CI中稳定运行（无flaky test）
- [ ] 测试数据是否独立管理（不依赖外部状态）
- [ ] 测试环境是否可复现（容器化/fixture管理）
- [ ] 测试执行时间是否在可接受范围内（单元<5min，集成<15min）
- [ ] 失败测试是否有清晰的错误信息和定位信息
- [ ] 是否有测试报告自动生成和归档

### 缺陷管理审查

- [ ] 所有缺陷是否按标准模板填写
- [ ] 缺陷等级是否正确分类
- [ ] P0/P1缺陷是否在SLA内修复
- [ ] 已修复缺陷是否有对应的回归测试用例
- [ ] 缺陷重开率是否低于10%
- [ ] 是否定期分析缺陷分布趋势并推动改进

## 交叉审查要点

### 与架构师交叉

- 测试策略是否覆盖架构中的所有集成点
- 是否对架构中标识的高风险组件重点测试
- 可测试性是否在架构设计阶段纳入考量

### 与安全专家交叉

- 安全测试用例是否覆盖威胁模型中的所有威胁
- 渗透测试是否纳入测试计划
- 安全扫描工具是否集成到CI流水线

### 与DevOps交叉

- 测试环境是否与生产环境配置一致
- CI流水线中测试阶段的超时和重试配置是否合理
- 测试报告是否自动归档和通知

### 与开发交叉

- 单元测试是否由开发编写并纳入代码提交标准
- 代码可测试性是否满足（依赖注入、接口隔离）
- 测试失败时开发响应流程是否明确

## 常见反模式

| 反模式 | 问题 | 正确做法 |
|-------|------|---------|
| 冰淇淋锥（E2E过多） | 执行慢、维护难、反馈延迟 | 遵循测试金字塔，增加单元测试 |
| 测试用例无断言 | 测试永远通过，无法发现问题 | 每个测试必须有明确断言 |
| 共享可变测试数据 | 测试顺序依赖，并行执行失败 | 每个测试独立创建数据 |
| 忽视Flaky测试 | 降低团队对测试结果的信任 | 立即修复或隔离不稳定测试 |
| 覆盖率崇拜 | 追求数字而非质量 | 关注分支覆盖和变异测试 |
| 测试代码不维护 | 测试与代码不同步 | 测试代码与生产代码同等对待 |
| 只测试正常路径 | 异常场景导致生产事故 | 负面测试至少占30% |
| 延迟测试到最后 | 缺陷修复成本倍增 | 测试左移，开发阶段即测试 |

## 知识依据

- **测试金字塔**：Martin Fowler《测试金字塔》、Google Testing Blog
- **等价类/边界值**：IEEE 829 测试文档标准、ISTQB基础认证大纲
- **质量门禁**：Google SRE《站点可靠性工程》错误预算模型
- **自动化策略**：Test Automation University、Playwright官方最佳实践
- **缺陷管理**：IEEE 1044 缺陷分类标准
- **度量指标**：DORA Metrics（部署频率、变更前置时间、故障恢复时间、变更失败率）
