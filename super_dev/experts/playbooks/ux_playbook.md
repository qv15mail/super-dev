# UX设计师（UX）操作手册

## 概述

UX设计师负责确保产品易用、高效、令人愉悦。核心职责包括用户旅程设计、信息架构规划、导航系统设计、交互规范制定、状态系统设计、可访问性保障。UX不是让界面好看，而是让用户在达成目标的路径上毫无阻碍。

### 核心原则

1. **用户目标优先**：每个设计决策服务于用户目标，而非产品指标
2. **减少认知负荷**：信息分层展示，避免一次性呈现过多选择
3. **一致性和可预期**：相同操作产生相同结果，用户可以建立心理模型
4. **容错设计**：允许犯错、易于撤销、提供清晰的错误恢复路径

## 方法论

### 一、用户旅程设计

#### 1.1 旅程地图框架

```
旅程阶段:
  认知 → 考虑 → 注册 → 首次使用 → 日常使用 → 高级使用 → 推荐/流失

每个阶段记录:
  用户目标: 用户在这个阶段想要什么
  触点: 用户与产品接触的方式（页面/邮件/通知）
  行为: 用户实际执行的操作
  情绪: 用户的感受（满意/困惑/沮丧）
  痛点: 阻碍用户的问题
  机会: 可以改善体验的切入点
```

#### 1.2 关键旅程识别

```
核心旅程（必须无摩擦）:
  1. 首次注册 → 首次成功使用（<= 5 步）
  2. 核心任务完成路径（如：下单/创建项目/发布内容）
  3. 错误恢复路径（如：支付失败后重试）

评估标准:
  完成时间: 核心任务是否可在3分钟内完成
  步骤数:   是否有可合并或省略的步骤
  分支点:   是否有不必要的决策点
  退出点:   用户最可能在哪一步放弃
```

#### 1.3 首次上手路径（FTUE）

```
设计要点:
  1. 注册流程 <= 3步（必填字段最小化）
  2. 注册后立即引导用户完成一个有价值的操作
  3. 渐进式引导（不一次性展示所有功能）
  4. 提供跳过选项（尊重熟练用户）
  5. 空状态提供明确的下一步行动

衡量指标:
  注册完成率
  首次关键操作完成率
  注册到活跃用户转化时间
```

### 二、信息架构规划

#### 2.1 架构设计原则

```
LATCH 组织原则:
  L - Location（位置）: 按地理/空间组织
  A - Alphabet（字母）: 按字母/拼音排序
  T - Time（时间）: 按时间线组织
  C - Category（分类）: 按类别组织
  H - Hierarchy（层级）: 按重要性/量级组织

选择依据:
  用户是否知道确切名称 → 字母排序
  信息有时间属性 → 时间线
  用户按功能区分 → 分类
  信息有天然包含关系 → 层级
```

#### 2.2 页面层级规划

```
层级结构（不超过3层）:
  L1 - 一级页面: 主要功能入口（首页/仪表盘/工作台）
  L2 - 二级页面: 功能列表/详情（项目列表/设置页）
  L3 - 三级页面: 具体操作/编辑（创建表单/详情编辑）

每层职责:
  L1: 全局导航 + 关键数据概览 + 快捷入口
  L2: 内容列表 + 筛选排序 + 批量操作
  L3: 表单/编辑 + 保存/取消 + 回到上一层

超过3层的处理:
  使用面板/抽屉/弹窗覆盖在当前页面
  使用面包屑导航保持位置感知
  考虑是否信息架构需要重新组织
```

### 三、导航设计原则

#### 3.1 导航模式选择

| 导航模式 | 适用场景 | 层级支持 | 示例 |
|---------|---------|---------|------|
| 顶部导航 | 功能模块少（<=7）| 1-2层 | 企业官网 |
| 侧边导航 | 功能模块多、层级深 | 2-3层 | 管理后台 |
| 底部Tab | 移动端核心功能 | 1层（<=5个Tab）| 手机App |
| 面包屑 | 辅助导航，标识位置 | 不限 | 电商分类 |
| 搜索 | 内容量大，用户知道要找什么 | 不限 | 文档站 |

#### 3.2 导航设计规范

```
深度限制: 最多3层（超过3层用户会迷失）
当前位置: 始终明确标识用户所在位置（高亮/面包屑）
可预见性: 导航标签应让用户预期点击后看到什么
一致性:   全局导航在所有页面保持不变

标签命名:
  使用用户语言而非系统术语
  2-4个字为佳，动宾结构（如"创建项目"）
  避免缩写和专业术语

移动端适配:
  汉堡菜单仅作为最后手段（可发现性差）
  优先使用底部Tab（核心功能）+ 更多按钮
  侧滑手势返回上一级
```

### 四、表单设计规范

#### 4.1 表单布局

```
单列布局: 默认选择（减少眼球跳跃，适合手机端）
多列布局: 仅当字段有逻辑分组且屏幕足够宽时使用

字段排列:
  1. 最常填写的字段在前
  2. 相关字段分组（个人信息/地址/支付各一组）
  3. 可选字段放在最后或折叠
  4. 操作按钮（提交/取消）在表单末尾

标签位置:
  顶部标签（推荐）: 扫描效率高，适合移动端
  左侧标签: 适合桌面端宽表单，字段较多时使用
  浮动标签: 节省空间，但可访问性需额外处理
```

#### 4.2 实时验证

```
验证时机:
  输入完成时（onBlur）: 优先使用，不打断输入流
  实时（onChange）: 仅用于格式类验证（手机号/邮箱格式提示）
  提交时: 作为最终兜底，但不应是唯一验证

验证规则:
  必填: 失焦后为空时提示"请输入{字段名}"
  格式: 即时提示格式要求（如"请输入有效的邮箱地址"）
  长度: 超出时限制输入或提示
  关联: 确认密码与密码不一致时提示

反模式:
  - 提交后才显示所有错误（用户需重新定位）
  - 输入过程中实时标红（造成焦虑）
  - 仅用红色标识错误（色盲用户无法识别）
```

#### 4.3 错误提示设计

```
错误提示规范:
  位置: 紧跟在出错字段下方
  颜色: 红色文字 + 红色边框（双通道，不仅靠颜色）
  图标: 前置警告图标（辅助色盲用户）
  措辞: 说明错误原因 + 如何修复

好的错误提示:
  "密码长度至少8位，需包含字母和数字"
  "此邮箱已注册，是否直接登录？"

差的错误提示:
  "格式不正确"
  "Error: validation failed"
  "请输入正确的值"
```

### 五、状态设计

#### 5.1 四种关键状态

```
空状态:
  场景: 列表无数据、搜索无结果、首次使用
  设计:
    - 简洁的插图或图标
    - 一句话说明为什么是空的
    - 提供明确的下一步行动按钮
    - 禁止只显示空白页面

加载状态:
  场景: 数据请求中、文件上传中、操作处理中
  设计:
    - < 1秒: 不显示加载指示（避免闪烁）
    - 1-3秒: 显示行内spinner或进度条
    - > 3秒: 显示骨架屏（skeleton）保持布局稳定
    - > 10秒: 提示预计等待时间或提供取消选项
    - 禁止全屏遮罩阻断所有操作

错误状态:
  场景: 网络异常、服务端错误、权限不足
  设计:
    - 说明发生了什么（用户语言）
    - 提供重试按钮（网络/超时类错误）
    - 提供替代方案（联系客服/返回上一页）
    - 保留用户已输入的数据（不清空表单）
    - 禁止只显示错误代码或技术堆栈

成功状态:
  场景: 操作完成、保存成功、提交成功
  设计:
    - 明确的成功反馈（对勾图标 + 文字）
    - 说明下一步可以做什么
    - 自动消失的Toast适合轻操作（3秒后消失）
    - 重要操作使用确认页面（如支付成功）
```

#### 5.2 状态转换

```
正常流: 空状态 → 加载中 → 有数据 → 操作 → 加载中 → 成功/错误
首次: 空状态 → 引导行动 → 加载中 → 有数据
网络异常: 有数据 → 操作 → 加载中 → 错误 → 重试 → 加载中 → 成功
离线: 有数据 → 离线提示横幅（保留已有数据展示）→ 恢复在线 → 自动同步
```

### 六、可访问性（WCAG 2.1）

```
A级（必须满足）:
  - 所有图片有 alt 文本
  - 所有表单输入有 label
  - 页面可以仅用键盘操作
  - 颜色不是传达信息的唯一手段

AA级（应当满足）:
  - 文本对比度 >= 4.5:1（大字 >= 3:1）
  - 可调整文字大小至200%不破坏布局
  - 焦点顺序符合逻辑
  - 错误提示可被辅助技术读取（aria-live）

键盘可访问性:
  Tab: 按逻辑顺序移动焦点
  Enter/Space: 触发按钮/链接
  Escape: 关闭弹窗/下拉
  方向键: 在列表/Tab之间移动
  焦点陷阱: 弹窗内焦点不逃逸到背景

ARIA 使用原则:
  1. 优先使用原生HTML语义元素（button/nav/main/aside）
  2. 无法使用原生元素时才使用ARIA角色
  3. 所有自定义组件必须有 role 和 aria-label
  4. 动态内容变化使用 aria-live 通知辅助技术
```

## UX审查检查清单

### 用户旅程审查

- [ ] 核心用户旅程是否已识别并文档化
- [ ] 首次使用路径是否 <= 5步到达首个价值点
- [ ] 注册流程是否最小化必填字段
- [ ] 是否有渐进式引导（不一次性展示所有功能）
- [ ] 关键路径上是否有不必要的中断或分支

### 信息架构审查

- [ ] 页面层级是否不超过3层
- [ ] 信息分类是否符合用户心理模型
- [ ] 相似功能是否聚合在一起
- [ ] 是否有重复或冗余的入口

### 导航审查

- [ ] 用户是否始终知道自己在哪里
- [ ] 导航标签是否使用用户语言
- [ ] 是否可以从任何页面回到首页
- [ ] 导航层级是否不超过3层
- [ ] 移动端导航是否可发现且易用

### 表单审查

- [ ] 是否使用单列布局（除非有充分理由）
- [ ] 是否有实时验证（onBlur优先）
- [ ] 错误提示是否说明了原因和修复方法
- [ ] 必填字段是否有明确标识
- [ ] 表单提交失败后是否保留用户输入
- [ ] 长表单是否分步展示

### 状态设计审查

- [ ] 空状态是否有引导行动
- [ ] 加载状态是否根据时长选择了合适的指示方式
- [ ] 错误状态是否提供了重试或替代方案
- [ ] 成功状态是否提示了下一步
- [ ] 是否处理了离线场景

### 可访问性审查

- [ ] 文本对比度是否 >= 4.5:1
- [ ] 所有图片是否有alt文本
- [ ] 页面是否可仅用键盘操作
- [ ] 焦点顺序是否符合逻辑
- [ ] 弹窗是否实现了焦点陷阱
- [ ] 动态内容是否使用aria-live
- [ ] 触控区域是否 >= 44px

## 交叉审查要点

### 与UI设计师交叉

- Token系统是否支撑了信息层级的视觉表达
- 组件状态设计是否与UX状态矩阵对齐
- 视觉焦点是否引导用户沿核心路径前进

### 与前端开发交叉

- 交互规范是否可实现（动画时长/缓动曲线/响应式行为）
- 状态管理是否覆盖了UX定义的所有状态
- 可访问性属性是否在代码中实现

### 与产品交叉

- 用户旅程是否匹配PRD中定义的核心场景
- 信息架构是否支持产品的扩展计划
- 关键指标（完成率/时长/满意度）是否可度量

### 与QA交叉

- 所有状态是否都有对应的测试用例
- 边界场景（超长文本/大量数据/慢网络）是否有设计方案
- 可访问性测试是否纳入QA流程

## 常见反模式

| 反模式 | 问题 | 正确做法 |
|-------|------|---------|
| 信息过载 | 一页展示所有功能 | 渐进式展示，按需呈现 |
| 导航迷宫 | 层级过深，用户迷失 | 不超过3层，面包屑辅助 |
| 空状态空白 | 新用户不知道下一步 | 提供引导和行动按钮 |
| 模态框滥用 | 打断用户操作流 | 仅用于确认不可逆操作 |
| 只靠颜色传达 | 色盲用户无法理解 | 颜色+图标+文字多通道 |
| 加载无反馈 | 用户以为页面卡死 | 适时的加载指示器 |
| 错误不可恢复 | 表单清空/无重试 | 保留数据+提供重试 |
| 触控区域过小 | 手指无法准确点击 | 最小44px可触控区域 |

## 知识依据

- **用户旅程**：JTBD（Jobs to Be Done）框架、Design Sprint方法论
- **信息架构**：Richard Saul Wurman《Information Architecture》、Card Sorting方法
- **导航设计**：Steve Krug《Don't Make Me Think》
- **表单设计**：Luke Wroblewski《Web Form Design》、Nielsen Norman Group表单指南
- **状态设计**：Scott Hurff《Designing Products People Love》五状态模型
- **可访问性**：W3C WCAG 2.1指南、WAI-ARIA Authoring Practices
- **启发式评估**：Jakob Nielsen十大可用性原则
