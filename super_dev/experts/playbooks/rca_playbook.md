# 根因分析专家（RCA）操作手册

## 概述

根因分析专家负责在故障发生后系统性地定位根本原因、推动修复并建立预防机制。核心职责包括故障响应协调、根因定位分析、时间线重建、改进措施制定、知识沉淀。RCA不是追责，而是通过科学方法找到系统性缺陷并消除再次发生的可能。

### 核心原则

1. **无责文化**：RCA聚焦系统和流程改进，不追究个人责任
2. **事实驱动**：所有结论基于日志、监控、配置等客观证据，不基于猜测
3. **深挖根因**：不停留在表面原因，直到找到可通过系统手段消除的根本原因
4. **闭环改进**：每个根因必须对应具体的改进行动项，并有明确责任人和截止日期

## 方法论

### 一、故障响应流程

#### 1.1 五阶段响应模型

```
检测 → 分类 → 缓解 → 修复 → 复盘

阶段1：检测（Detection）
  触发源: 监控告警 / 用户反馈 / 内部发现
  目标: 确认故障存在，记录首次发现时间
  输出: 故障通报（时间/现象/影响范围初判）

阶段2：分类（Classification）
  依据: 影响范围 x 影响程度
  P0: 全量用户核心功能不可用 → 全员响应
  P1: 部分用户或核心功能降级 → 主要团队响应
  P2: 非核心功能异常 → 相关团队响应
  P3: 轻微异常/用户少量感知 → 常规处理
  输出: 故障等级 + 响应团队 + 沟通渠道

阶段3：缓解（Mitigation）
  目标: 最快速度恢复服务（不一定是根本修复）
  常用手段:
    - 回滚到上一个稳定版本
    - 切换到备用节点/区域
    - 关闭触发故障的功能开关
    - 扩容/重启受影响服务
    - 流量切走有问题的节点
  输出: 服务恢复确认 + 临时措施记录

阶段4：修复（Fix）
  目标: 彻底修复根本原因
  流程:
    1. 定位根因（使用下述分析方法）
    2. 开发修复方案
    3. 在预发布环境验证
    4. 灰度发布修复
    5. 全量发布并监控
  输出: 修复代码 + 回归测试 + 部署确认

阶段5：复盘（Postmortem）
  时间: 故障修复后 48 小时内
  参与: 所有相关团队
  输出: RCA报告 + 行动项清单
```

#### 1.2 故障通报模板

```
【故障通报】{等级} - {简述}
时间: {首次检测时间}
影响: {受影响的用户/功能/数据范围}
当前状态: 处理中 / 已缓解 / 已修复
负责人: {oncall/指挥官}
沟通渠道: {群聊/频道/电话桥}
下次更新: {预计时间}
```

### 二、5 Whys 分析法

#### 2.1 方法说明

从表面现象出发，反复追问"为什么"，直到找到可以通过系统手段消除的根本原因。通常需要3-7层追问。

#### 2.2 完整示例

```
现象: 用户报告下午14:30无法访问网站

Why 1: 为什么用户无法访问？
  → Nginx返回502 Bad Gateway

Why 2: 为什么Nginx返回502？
  → 后端API服务全部无响应

Why 3: 为什么API服务无响应？
  → API服务因OOM被Kubernetes杀掉，且无法重新启动

Why 4: 为什么API服务OOM？
  → 一个批量导出请求将全部数据加载到内存中，单请求消耗了8GB内存

Why 5: 为什么单个请求能消耗8GB内存？
  → 导出接口没有分页，没有内存限制，也没有请求大小保护

根因: 批量导出接口缺少分页处理和内存保护机制

改进措施:
  1. [短期] 导出接口添加流式处理和分页（MUST）
  2. [短期] 设置单个请求内存上限（MUST）
  3. [中期] 为所有查询类接口添加结果集大小限制（SHOULD）
  4. [长期] 建立内存使用的自动化测试和CI检查（SHOULD）
```

#### 2.3 5 Whys 注意事项

```
避免:
  - 停留在人为错误（"因为开发者忘了"不是根因，应继续问为什么流程没有检测出来）
  - 发散到多个分支而不收敛
  - 答案变成循环引用

技巧:
  - 每一层"Why"都要有客观证据支撑
  - 当答案涉及人为操作时，继续追问流程/工具为什么没有防护
  - 根因应该是可以通过技术/流程手段消除的系统性原因
```

### 三、鱼骨图方法（石川图）

#### 3.1 六大分类

```
                        +-- 人员 (People)
                        |    - 经验不足 / 操作失误 / 培训缺失
                        +-- 流程 (Process)
                        |    - 审查遗漏 / 流程缺失 / 变更管理不当
故障现象 <--------------+-- 技术 (Technology)
                        |    - 代码缺陷 / 架构瓶颈 / 依赖问题
                        +-- 环境 (Environment)
                        |    - 配置错误 / 资源不足 / 网络异常
                        +-- 工具 (Tools)
                        |    - 监控盲区 / 告警缺失 / 部署工具故障
                        +-- 外部 (External)
                             - 第三方服务故障 / 流量突增 / DNS问题
```

#### 3.2 使用步骤

```
1. 在右侧写下故障现象（明确、具体）
2. 画出六条主分支（六大分类）
3. 团队成员逐一头脑风暴，将可能原因归入对应分支
4. 对每个可能原因追问 Why，深入细化
5. 用证据验证或排除每个可能原因
6. 圈出经验证的根因
7. 为每个根因制定改进措施
```

### 四、时间线构建

#### 4.1 时间线模板

```
时间线格式:
  {时间} | {来源} | {事件描述}

示例:
  14:00 | 部署系统 | 版本v2.3.1开始灰度发布（5%流量）
  14:10 | 监控系统 | 错误率从0.1%上升到0.3%（未触发告警阈值）
  14:15 | 部署系统 | 灰度扩大到25%流量
  14:20 | 监控系统 | 错误率上升到2.5%，P0告警触发
  14:22 | oncall   | 收到告警，开始排查
  14:25 | oncall   | 确认为v2.3.1引入的问题，开始回滚
  14:28 | 部署系统 | 回滚到v2.3.0完成
  14:30 | 监控系统 | 错误率恢复到0.1%
  14:35 | oncall   | 确认服务恢复，发布故障恢复通报
```

#### 4.2 时间线数据来源

```
必须收集:
  - 部署记录（谁/什么时间/部署了什么版本）
  - 配置变更记录
  - 监控指标（错误率/延迟/流量/资源使用）
  - 告警触发记录
  - 值班人员操作记录
  - 用户反馈时间线
  - 日志（应用日志/系统日志/审计日志）

构建原则:
  - 所有时间使用统一时区（UTC或公司标准时区）
  - 精确到分钟（关键操作精确到秒）
  - 区分事实（日志记录）和推断（需标注）
  - 包含"什么没发生"（如：告警为什么没触发）
```

### 五、行动项制定（SMART原则）

#### 5.1 SMART标准

```
S - Specific（具体）: 明确做什么、改什么
  Bad:  "改善监控"
  Good: "为支付服务添加P99延迟告警，阈值500ms，持续3分钟触发"

M - Measurable（可衡量）: 有明确的完成标准
  Bad:  "提高代码质量"
  Good: "核心模块单元测试覆盖率从60%提升到85%"

A - Assignable（可分配）: 有明确的责任人
  Bad:  "团队一起解决"
  Good: "由@张三负责，@李四review"

R - Realistic（可实现）: 在给定资源和时间内可完成
  Bad:  "重写整个支付系统"（太大）
  Good: "为支付接口添加重试和幂等保护"

T - Time-bound（有时限）: 明确截止日期
  Bad:  "尽快完成"
  Good: "2024-02-15前上线"
```

#### 5.2 行动项分类

```
紧急修复（Immediate Fix）:
  时限: 24小时内
  内容: 直接消除本次故障的原因
  示例: 修复导致OOM的SQL查询

短期改进（Short-term Improvement）:
  时限: 1-2周
  内容: 加固防护，防止同类问题
  示例: 添加查询结果集大小限制、添加告警

中期改进（Medium-term Improvement）:
  时限: 1-2月
  内容: 系统性解决根本原因
  示例: 重构批量操作架构，引入流式处理

长期改进（Long-term Improvement）:
  时限: 1个季度
  内容: 预防性改进，提升整体可靠性
  示例: 建立混沌工程实践，定期故障演练
```

### 六、预防性改进措施

```
技术层面:
  - 添加缺失的监控和告警
  - 增加自动化测试覆盖（尤其是故障场景）
  - 引入混沌工程（故障注入测试）
  - 改进架构（限流/熔断/降级）

流程层面:
  - 完善变更管理流程（审批/灰度/回滚）
  - 建立checklist（部署前/上线后检查项）
  - 定期故障演练（每季度至少一次）
  - 知识库沉淀（故障案例/处理手册/runbook）

工具层面:
  - 完善监控仪表盘（覆盖四黄金信号）
  - 自动化告警升级（告警→值班→主管→CTO）
  - 一键回滚工具
  - 故障通报自动化
```

## RCA检查清单

### 故障响应审查

- [ ] 故障是否被及时检测（检测延迟是否可接受）
- [ ] 故障分级是否准确
- [ ] 缓解措施是否在SLA内执行
- [ ] 故障通报是否及时发出
- [ ] 通报内容是否完整（时间/影响/状态/负责人）
- [ ] 沟通渠道是否畅通

### 根因分析审查

- [ ] 5 Whys是否追问到了系统性根因（而非人为错误）
- [ ] 每层Why是否有客观证据支撑
- [ ] 是否考虑了多个贡献因素（不仅仅是触发因素）
- [ ] 时间线是否完整重建
- [ ] 时间线是否使用了统一时区
- [ ] 是否收集了所有相关数据源（日志/监控/部署记录）

### 行动项审查

- [ ] 每个根因是否有对应的行动项
- [ ] 行动项是否符合SMART原则
- [ ] 是否区分了紧急/短期/中期/长期
- [ ] 每个行动项是否有明确责任人
- [ ] 每个行动项是否有截止日期
- [ ] 行动项是否有跟踪机制（不会遗忘）

### 预防性审查

- [ ] 类似问题是否可以被自动检测
- [ ] 是否添加了缺失的监控和告警
- [ ] 是否有回归测试覆盖本次故障场景
- [ ] 变更管理流程是否需要加固
- [ ] 是否安排了故障演练验证改进效果
- [ ] 经验教训是否沉淀到知识库

### 复盘质量审查

- [ ] 复盘是否在故障修复后48小时内完成
- [ ] 复盘是否所有相关方参与
- [ ] 报告是否遵循无责文化（不追究个人）
- [ ] RCA报告是否归档并可检索

## 交叉审查要点

### 与DevOps交叉

- 部署流水线是否有回滚能力
- 灰度发布是否有自动异常检测和回滚
- 监控和告警覆盖是否完整

### 与架构师交叉

- 架构是否有足够的容错设计（熔断/降级/限流）
- 单点故障是否已消除
- 故障影响是否可以被隔离

### 与QA交叉

- 故障场景是否纳入回归测试
- 是否需要增加混沌测试用例
- 性能测试是否覆盖了边界场景

### 与安全专家交叉

- 故障是否涉及安全事件
- 日志和证据是否保全（审计需要）
- 缓解措施是否引入了新的安全风险

## 常见反模式

| 反模式 | 问题 | 正确做法 |
|-------|------|---------|
| 追责文化 | 团队隐瞒问题，不敢报告 | 无责复盘，聚焦系统改进 |
| 浅层分析 | 停在"操作失误"就结束 | 继续追问流程/工具为什么没防住 |
| 行动项遗忘 | 复盘后无人跟进 | 行动项纳入Sprint，定期回顾 |
| 报告模板化 | 填表了事，无实质分析 | 要求证据支撑每个结论 |
| 只修不防 | 修了bug就完事 | 必须有预防性改进措施 |
| 信息孤岛 | RCA报告锁在团队内 | 知识库开放，全公司可检索 |
| 演练缺失 | 只在真实故障中学习 | 定期混沌工程和故障演练 |

- **5 Whys**：Toyota Production System、Taiichi Ohno《丰田生产方式》
- **鱼骨图**：Kaoru Ishikawa质量管理方法
- **无责文化**：Etsy《Blameless Postmortems》、John Allspaw的系统安全理论
- **混沌工程**：Netflix Chaos Engineering、Principles of Chaos Engineering
- **SMART原则**：Peter Drucker《管理的实践》
- **事件管理**：PagerDuty Incident Response文档、Atlassian Incident Management手册
