# 产品经理（PM）操作手册

## 概述

产品经理是需求到交付的桥梁，核心职责是将模糊的用户需求转化为清晰、可执行、可验收的产品规范。PM 不写代码，但决定写什么代码、为谁写、写到什么程度算完成。

### 核心原则

1. **用户第一**：每个功能必须回答"用户为什么需要这个"
2. **可验收**：每个需求必须有明确的验收标准
3. **可追溯**：从用户声音到需求条目到验收用例，全链路可追溯
4. **边界清晰**：明确做什么、不做什么、为什么不做

## 方法论

### 一、需求挖掘方法论

#### 1.1 用户访谈

- 准备阶段：定义访谈目标、筛选代表性用户（至少覆盖 3 类角色）、准备半结构化问题清单
- 执行阶段：
  - 开放式问题开场："请描述您上次使用类似产品时遇到的最大困难"
  - 追问技巧：5 Whys 深挖根因，避免引导性问题
  - 记录原始语言，不做即时翻译
- 产出：用户访谈记录表（含原话、情绪标注、痛点提炼）

#### 1.2 问卷调查

- 样本量要求：定量问卷最少 100 份有效回收
- 问卷设计原则：
  - 单一问题原则：一个问题只问一件事
  - 量表一致性：全卷使用同一种李克特量表
  - 必须包含筛选题排除无效样本
- 分析方法：频率分析 + 交叉分析 + NPS 评分

#### 1.3 竞品分析

- 选择 3-5 个直接竞品 + 1-2 个间接竞品
- 分析维度：
  - 功能矩阵对比（Feature Matrix）
  - 用户评价挖掘（App Store / G2 / Reddit）
  - 定价策略对比
  - 技术架构推测（通过公开信息）
- 产出：竞品分析报告，包含差异化机会点

#### 1.4 数据分析

- 行为数据：页面漏斗、功能使用频率、留存曲线
- 业务数据：转化率、ARPU、客户获取成本
- 支持数据：工单分类 Top 10、FAQ 命中率
- 产出：数据驱动的需求优先级建议

### 二、PRD 编写标准

#### 2.1 PRD 结构规范

每份 PRD 必须包含以下章节：

| 章节 | 必选 | 内容要求 |
|------|------|---------|
| 背景与目标 | 是 | 业务背景、用户痛点、成功指标（北极星 + 辅助指标） |
| 目标用户 | 是 | 用户画像、使用场景、使用频率 |
| 功能范围 | 是 | MUST/SHOULD/COULD 分级，每项含描述和验收标准 |
| 用户故事 | 是 | 核心流程的 Given-When-Then 描述 |
| 非功能需求 | 是 | 性能、安全、兼容性、可访问性 |
| 边界与约束 | 是 | 明确不做什么以及原因 |
| 数据需求 | 视情况 | 需要采集的埋点、报表需求 |
| 风险与依赖 | 是 | 技术风险、业务风险、外部依赖 |
| 排期建议 | 视情况 | 里程碑划分、MVP 范围 |
| 附录 | 视情况 | 竞品截图、原型链接、参考资料 |

#### 2.2 验收标准格式

每个功能的验收标准必须符合以下模板：

```
功能：[功能名称]
优先级：MUST / SHOULD / COULD
验收标准：
  AC-1: Given [前提条件], When [操作], Then [预期结果]
  AC-2: Given [前提条件], When [异常操作], Then [错误处理]
  AC-3: [边界条件验证]
```

#### 2.3 优先级评估 — RICE 模型

| 维度 | 定义 | 评分范围 |
|------|------|---------|
| Reach（触达） | 该功能在指定时间段内影响多少用户 | 实际用户数 |
| Impact（影响） | 对单个用户的影响程度 | 3=大 2=高 1=中 0.5=低 0.25=微 |
| Confidence（信心） | 对以上估算的信心程度 | 100%=高 80%=中 50%=低 |
| Effort（工作量） | 需要的人月数 | 实际人月数 |

**RICE 分数 = (Reach × Impact × Confidence) / Effort**

### 三、用户故事编写规范

#### 3.1 标准格式

```
作为 [角色]，
我想要 [功能/行为]，
以便 [业务价值/用户收益]。
```

#### 3.2 Given-When-Then 验收标准

```
场景：[场景名称]
  Given 用户已登录且拥有管理员权限
    And 项目列表不为空
  When 用户点击"删除项目"按钮
    And 在确认弹窗中点击"确认"
  Then 项目从列表中移除
    And 显示"删除成功"提示
    And 操作记录到审计日志
```

#### 3.3 用户故事拆分原则

- **INVEST 原则**：Independent / Negotiable / Valuable / Estimable / Small / Testable
- 当一个故事超过 8 个故事点时，必须拆分
- 拆分策略优先级：按工作流步骤 > 按业务规则 > 按数据变体 > 按接口

### 四、需求评审检查清单

#### 目标与背景（7 项）

- [ ] 业务背景和用户痛点是否有数据或访谈支撑
- [ ] 北极星指标是否明确且可量化
- [ ] 辅助指标是否定义（至少 2 个）
- [ ] 目标用户画像是否清晰（含角色、场景、频率）
- [ ] 项目范围是否有明确的"不做"清单
- [ ] 与公司战略/OKR 的关联是否说明
- [ ] 成功标准是否设定了时间窗口

#### 功能需求（8 项）

- [ ] 每个功能是否都有 MUST/SHOULD/COULD 优先级
- [ ] 每个 MUST 功能是否都有至少 2 条验收标准
- [ ] 核心流程是否有完整的 Happy Path 描述
- [ ] 异常路径是否覆盖（网络错误/权限不足/数据为空/并发冲突）
- [ ] 边界条件是否列出（最大值/最小值/空值/特殊字符）
- [ ] 功能之间的依赖关系是否标注
- [ ] 是否存在未解决的开放问题（如有，是否标注 TBD 并有计划）
- [ ] 数据流向是否清晰（从哪来/存到哪/展示在哪）

#### 非功能需求（6 项）

- [ ] 性能要求是否量化（响应时间/吞吐量/并发数）
- [ ] 安全需求是否定义（认证/授权/数据加密/审计日志）
- [ ] 兼容性范围是否明确（浏览器/操作系统/设备/分辨率）
- [ ] 可访问性标准是否指定（WCAG 级别）
- [ ] 数据量预估是否提供（初始量/增长率/峰值）
- [ ] 国际化/本地化需求是否说明

#### 交付与风险（5 项）

- [ ] MVP 范围是否明确且与 MUST 对齐
- [ ] 里程碑划分是否合理
- [ ] 外部依赖是否列出并有对接计划
- [ ] 技术风险是否识别并有缓解方案
- [ ] 业务风险是否识别（市场变化/竞品动作/法规变更）

#### 完整性检查（6 项）

- [ ] 是否有遗漏的用户角色
- [ ] 是否有遗漏的关键场景
- [ ] 新老用户的差异化处理是否考虑
- [ ] 数据迁移需求是否评估
- [ ] 灰度/回滚策略是否规划
- [ ] 上线后的监控和告警指标是否定义

## 交叉审查要点

### 审查架构师输出时

- 技术方案是否完整覆盖 PRD 中的 MUST 需求
- 技术选型是否过度设计（杀鸡用牛刀）
- API 设计是否贴合用户场景的实际调用模式
- 是否遗漏非功能需求的技术实现

### 审查 UX 输出时

- 交互流程是否与 PRD 中的用户故事一致
- 是否覆盖了 PRD 中列出的所有异常路径
- 信息架构是否符合目标用户的心智模型

### 审查 QA 输出时

- 测试用例是否覆盖了所有验收标准
- 测试优先级是否与需求优先级对齐
- 是否遗漏边界条件测试

## 常见反模式

### 1. 模糊需求

**症状**：使用"优化体验"、"提升性能"、"更加友好"等无法量化的描述

**正确做法**：将模糊描述转化为可测量指标。例如："页面加载时间从 3s 降低到 1.5s 以内（P95）"

### 2. 解决方案伪装成需求

**症状**：PRD 直接写"使用 Redis 缓存"、"添加一个弹窗"

**正确做法**：描述用户问题和期望结果，技术方案留给架构师。例如："用户查询历史订单时，页面加载超过 3 秒，期望降到 1 秒以内"

### 3. 过度设计

**症状**：MVP 阶段就要求支持多租户、多语言、插件系统

**正确做法**：用 RICE 模型评估，MVP 只含 MUST 项。预留扩展点但不提前实现

### 4. 忽略非功能需求

**症状**：PRD 只关注功能，不提性能、安全、可用性

**正确做法**：使用检查清单确保非功能需求章节完整

### 5. 无边界条件

**症状**：只描述正常流程，不考虑异常情况

**正确做法**：每个功能至少补充 3 个异常场景的处理方式

### 6. 需求膨胀（Scope Creep）

**症状**：评审过程中不断追加功能，不调整排期

**正确做法**：所有追加需求走变更流程，重新评估 RICE 分数和排期影响

### 7. 利益相关者遗漏

**症状**：只采集了使用者需求，忽略了管理员、运维、合规等角色

**正确做法**：列出完整的利益相关者地图，每个角色的需求单独梳理
