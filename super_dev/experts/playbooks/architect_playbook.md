# 架构师（ARCHITECT）操作手册

## 概述

架构师负责将产品需求转化为可落地的技术方案，核心职责包括系统边界定义、模块拆分、技术选型、通信契约设计和架构决策记录。架构师不追求理论完美，追求在约束条件下做出可演进的最优决策。

### 核心原则

1. **约束驱动**：技术选型必须基于业务约束（团队规模、交付时间、运维能力），而非技术偏好
2. **可演进**：当前方案必须为未来 6-12 个月的演进留出扩展点
3. **可验证**：每个架构决策都应有验证手段（PoC/基准测试/容量估算）
4. **有记录**：关键决策必须留下 ADR，降低后续返工和知识流失风险

## 方法论

### 一、系统设计方法论

#### 1.1 需求分析阶段

- 从 PRD 提取功能需求，映射到技术能力要求
- 提取非功能需求，量化到具体数值：
  - QPS 峰值、P99 延迟、可用性 SLA
  - 数据量（初始/年增长/峰值）
  - 并发用户数（日常/峰值）
- 识别关键业务场景，排列设计优先级
- 列出已有技术栈约束和团队能力边界

#### 1.2 高层设计阶段

- 绘制系统上下文图（System Context Diagram）
  - 标注所有外部系统和用户角色
  - 标注数据流方向和协议
- 确定系统拆分策略：
  - 单体优先（团队 < 5 人，模块 < 10 个）
  - 模块化单体（团队 5-15 人）
  - 微服务（团队 > 15 人，且有明确的独立部署需求）
- 定义模块边界和职责矩阵
- 确定通信模式（同步 REST/gRPC vs 异步消息队列）

#### 1.3 详细设计阶段

- API 契约设计（OpenAPI/Protobuf）
- 数据模型设计（与 DBA 协作）
- 关键流程时序图
- 容量估算与扩展策略
- 错误处理和降级策略
- 安全设计（与安全专家协作）

#### 1.4 评审阶段

- 内部架构评审（团队内）
- 跨团队评审（有依赖时）
- 产出 ADR 并归档

### 二、技术选型决策框架

#### 2.1 评估矩阵

| 维度 | 权重 | 评分标准（1-5） |
|------|------|---------------|
| 功能适配度 | 25% | 是否满足当前需求和 6 个月内的扩展需求 |
| 团队熟悉度 | 20% | 团队是否有实际生产经验 |
| 社区活跃度 | 15% | GitHub star/issue 响应/版本频率/Stack Overflow 活跃度 |
| 运维成本 | 15% | 部署复杂度/监控支持/故障排查便利性 |
| 性能表现 | 10% | 基准测试结果是否满足非功能需求 |
| 许可证合规 | 10% | 是否与项目许可证兼容 |
| 长期风险 | 5% | 供应商锁定程度/替代方案可用性 |

#### 2.2 ADR（架构决策记录）模板

```markdown
# ADR-{编号}: {决策标题}

## 状态
提议 / 已接受 / 已弃用 / 已取代

## 背景
[什么业务或技术需求驱动了这个决策]

## 决策
[我们选择了什么方案]

## 备选方案
### 方案 A: [名称]
- 优点：...
- 缺点：...

### 方案 B: [名称]
- 优点：...
- 缺点：...

## 理由
[为什么选择当前方案而非其他方案]

## 影响
[这个决策带来的正面和负面影响]

## 验证方式
[如何验证这个决策是否正确]
```

#### 2.3 选型维度详解

**数据库选型**：
- OLTP 场景：PostgreSQL（通用）、MySQL（高并发读场景）
- 文档型：MongoDB（schema 频繁变化）
- 缓存：Redis（通用缓存/会话）
- 搜索：Elasticsearch（全文搜索/日志）
- 时序：InfluxDB/TimescaleDB（监控/IoT）

**消息队列选型**：
- 通用消息：RabbitMQ（功能丰富）、Kafka（高吞吐）
- 任务队列：Celery（Python）、Bull（Node.js）
- 事件流：Kafka（持久化+重放）

**API 协议选型**：
- 面向前端：REST（通用）、GraphQL（查询灵活/减少请求）
- 服务间通信：gRPC（高性能/强类型）、REST（简单场景）
- 实时通信：WebSocket（双向）、SSE（服务端推送）

### 三、API 设计规范

#### 3.1 RESTful 规范

**URL 设计**：
```
GET    /api/v1/projects              # 列表查询
POST   /api/v1/projects              # 创建
GET    /api/v1/projects/{id}         # 获取详情
PUT    /api/v1/projects/{id}         # 全量更新
PATCH  /api/v1/projects/{id}         # 部分更新
DELETE /api/v1/projects/{id}         # 删除
GET    /api/v1/projects/{id}/tasks   # 嵌套资源
```

**命名规范**：
- URL 使用 kebab-case：`/api/v1/user-profiles`
- 查询参数使用 snake_case：`?sort_by=created_at&page_size=20`
- JSON 字段使用 camelCase（前端友好）或 snake_case（与后端一致），全项目统一

**分页规范**：
```json
{
  "data": [...],
  "pagination": {
    "page": 1,
    "page_size": 20,
    "total_count": 150,
    "total_pages": 8
  }
}
```

#### 3.2 版本控制

- URL 路径版本号：`/api/v1/`、`/api/v2/`
- 主版本号在有 Breaking Change 时递增
- 旧版本至少保留 6 个月并在响应头标注 Deprecation

#### 3.3 统一错误码

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "请求参数不合法",
    "details": [
      {"field": "email", "reason": "格式不正确"}
    ],
    "request_id": "req_abc123"
  }
}
```

HTTP 状态码使用规范：
| 状态码 | 场景 |
|-------|------|
| 200 | 成功（GET/PUT/PATCH） |
| 201 | 创建成功（POST） |
| 204 | 删除成功（DELETE） |
| 400 | 请求参数错误 |
| 401 | 未认证 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 资源冲突（并发/重复） |
| 422 | 业务规则校验失败 |
| 429 | 限流 |
| 500 | 服务器内部错误 |
| 503 | 服务不可用 |

#### 3.4 gRPC 规范

- Proto 文件按服务拆分，放入 `proto/` 目录
- 使用 `google.api.http` 注解实现 REST 兼容
- 错误处理使用 `google.rpc.Status`
- 超时必须在 client 端设置，默认 5 秒

### 四、数据流设计

#### 4.1 设计流程

1. **ER 图**：实体关系建模，确定核心实体和关系类型
2. **数据流图**：标注数据从输入到存储到展示的全路径
3. **API 契约**：基于数据流确定每个 API 的请求/响应结构
4. **缓存策略**：识别读多写少的热点数据，设计缓存层级

#### 4.2 缓存设计

- L1 缓存：进程内缓存（LRU，适合配置类数据）
- L2 缓存：Redis（分布式缓存，适合会话/热点数据）
- 缓存策略：Cache-Aside（通用）、Read-Through、Write-Behind（写密集）
- 缓存失效：TTL + 主动失效（数据变更时）
- 缓存穿透防护：布隆过滤器 / 空值缓存
- 缓存雪崩防护：随机 TTL + 互斥锁

## 审查检查清单

### 系统边界与模块（8 项）

- [ ] 系统上下文图是否绘制，外部系统和用户角色是否标注完整
- [ ] 模块职责是否单一且边界清晰
- [ ] 模块间通信方式是否明确（同步/异步/事件）
- [ ] 模块间是否存在循环依赖
- [ ] 共享数据库是否有清晰的所有权划分
- [ ] 是否存在跨模块的直接数据库访问
- [ ] 模块拆分粒度是否匹配团队规模
- [ ] 是否考虑了模块的独立部署和独立扩展能力

### API 与契约（6 项）

- [ ] API 是否有 OpenAPI/Protobuf 定义
- [ ] 是否定义了版本控制策略
- [ ] 错误码是否统一且有文档
- [ ] 分页/排序/过滤是否有统一规范
- [ ] API 幂等性是否考虑（POST/PUT 操作）
- [ ] API 限流策略是否定义

### 可靠性与扩展性（8 项）

- [ ] 关键路径是否做了容量估算
- [ ] 单点故障是否识别并有冗余方案
- [ ] 数据库读写分离是否评估
- [ ] 缓存策略是否设计（含失效/穿透/雪崩应对）
- [ ] 异步处理是否有重试和死信队列
- [ ] 是否有降级和熔断策略
- [ ] 超时设置是否合理（调用链路每层递减）
- [ ] 是否考虑了水平扩展的无状态设计

### 安全性（5 项）

- [ ] 认证方式是否确定（JWT/OAuth2/API Key）
- [ ] 授权模型是否设计（RBAC/ABAC）
- [ ] 敏感数据是否加密传输和存储
- [ ] 日志中是否脱敏个人信息
- [ ] 跨域策略是否配置

### 可观测性（5 项）

- [ ] 日志规范是否统一（格式/级别/关联 ID）
- [ ] 分布式追踪是否接入（Jaeger/Zipkin）
- [ ] 核心指标是否暴露（延迟/错误率/吞吐量/饱和度）
- [ ] 健康检查端点是否提供
- [ ] 告警规则是否预定义

## 交叉审查要点

### 审查 PM 输出时

- 非功能需求是否量化到可以指导技术方案
- 数据量预估是否合理
- 是否有遗漏的系统集成需求

### 审查安全专家输出时

- 威胁模型是否覆盖了所有对外接口
- 安全需求是否可以在当前架构下落地
- 安全方案是否引入了不可接受的性能损耗

### 审查 DBA 输出时

- 数据模型是否与系统模块边界一致
- 索引策略是否覆盖了核心查询场景
- 数据库选型是否与架构方案匹配

### 审查 DevOps 输出时

- CI/CD 流水线是否匹配模块拆分策略
- 部署方案是否支持独立部署和回滚
- 监控方案是否覆盖架构中定义的关键指标

## 常见反模式

### 1. 分布式单体

**症状**：拆成了多个服务，但服务间强耦合，必须同时部署

**识别方式**：一个功能变更需要同时修改 3 个以上服务

**正确做法**：先做模块化单体，只在有明确的独立扩展/部署需求时才拆分

### 2. 过度设计

**症状**：MVP 阶段就引入 Kafka + Kubernetes + 微服务网格

**识别方式**：技术复杂度远超业务复杂度

**正确做法**：从最简方案开始，在瓶颈出现时再引入复杂组件

### 3. 缺少边界

**症状**：多个模块共享数据库表、直接调用内部方法

**识别方式**：无法独立测试单个模块

**正确做法**：每个模块通过定义好的接口交互，数据库表有明确的所有权

### 4. 无 ADR

**症状**：技术决策只在口头讨论，新成员不知道为什么选了某个方案

**识别方式**：团队成员对"为什么用 X 而不用 Y"无法给出一致回答

**正确做法**：每个关键决策写 ADR，记录背景、方案、理由和影响

### 5. 简历驱动开发

**症状**：选型基于"我想学这个技术"而非业务需求

**识别方式**：选型评估矩阵中"团队熟悉度"和"运维成本"得分很低

**正确做法**：严格使用评估矩阵，团队熟悉度权重不低于 20%

### 6. 忽略运维成本

**症状**：方案在开发环境完美运行，但生产环境运维极其困难

**识别方式**：没有考虑监控、日志、告警、故障恢复

**正确做法**：架构设计必须包含可观测性章节，运维成本纳入选型评估
