# Super Dev SEEAI - 赛事行动模式

> 赛事专用执行循环 | 不走标准流水线 | 行动优先

---

## 定位

SEEAI 赛事模式是 Super Dev 的独立指令，与标准 `/super-dev` 流程完全分离。
标准流程是治理驱动（文档→门禁→审查→交付），赛事模式是**行动驱动**。
触发 SEEAI 后全程留在赛事合同内，不切回标准长流程。

## 触发方式

- `/super-dev-seeai <需求>`
- `super-dev-seeai: <需求>`
- `super-dev-seeai：<需求>`

---

## 执行循环（强制）

```
需求 → 拆解 → 联网搜索 → 方案文档 → Spec → 写码 → 跑起来 → 等反馈
                                                         ↑              ↓
                                                         └── 改 ← 用户反馈
```

**六个阶段，没有门禁，没有确认等待（方案文档除外），一路执行到跑起来。**

---

## 阶段 1: 拆解需求

收到需求后，立刻拆解：

1. **这个东西本质是什么？** — 游戏 / 工具 / 展示页 / 可视化 / 应用
2. **核心体验是什么？** — 用户打开后最先感受到的那一件事
3. **必须做的（P0）** — 没有这个就不算完成
4. **加分项（P1）** — 有了会让作品亮眼
5. **不做的** — 明确放弃什么，不贪多

需求模糊时最多问用户**1个关键问题**，能假设就直接推进。

输出：直接在回复中列出以上 5 点，不生成文件。

---

## 阶段 2: 联网搜索

用宿主的 WebSearch / WebFetch 能力做两件事：

1. **找参考** — 搜索同类作品、类似产品，了解"好的"长什么样
2. **查技术** — 确认要用的框架/库的精确版本、API 写法、已知坑

搜索目标不是"调研报告"，而是为下一步的方案文档和 Spec 提供**可靠的技术依据**。

关键：如果选了某个库，必须确认它的 API 在当前版本下是正确的。不确认就不用。

---

## 阶段 3: 方案文档

写一个文件 `output/solution.md`，替代标准流程的 PRD + 架构 + UIUX 三文档。
这是赛事模式下**唯一需要用户过目的文档**。

### 方案文档结构

```markdown
# 方案：[作品名称]

## 做什么
一段话说清楚作品目标和核心体验。

## 怎么做
- 技术栈：[具体框架 + 版本]
- 项目结构：[几个关键文件/目录]
- 核心逻辑：[主循环/主流程是什么]
- 数据流：[数据从哪来、怎么流转、存在哪]

## 长什么样
- 视觉方向：[色调、风格、氛围]
- 首屏内容：[用户打开后看到什么]
- 关键交互：[主要操作和反馈]

## 优先级
- P0（必须完成）：...
- P1（加分项）：...
- 不做：...
```

写完后告知用户方案文档已生成，**等用户确认或反馈后再继续**。
用户说"好/确认/继续/可以"→ 进入 Spec。
用户给出修改意见 → 更新文档，再次等确认。

---

## 阶段 4: Spec（技术方案锁定）

Spec 的作用是**写代码前把所有技术决策想清楚**，减少编码阶段的返工。

写入 `output/spec.md`，包含：

### 4.1 依赖清单（精确版本）

```markdown
## 依赖
- xxx@1.2.3 — 用途，已确认 API 兼容
- yyy@4.5.6 — 用途，已确认 API 兼容
```

每个依赖必须在阶段 2 联网确认过版本和 API。**不确认不写进 Spec。**

### 4.2 文件清单（含 import 关系）

```markdown
## 文件结构
- src/main.js — 入口，import { xxx } from './game.js'
- src/game.js — 游戏核心，import { yyy } from './utils.js'
- src/utils.js — 工具函数，无外部依赖
- index.html — 页面入口，引用 main.js
```

每个文件的 import 来源必须明确。写代码时不需要猜 import 路径。

### 4.3 API / 接口约定

如果有后端或外部 API 调用，列出：
- 请求路径
- 请求方法
- 参数格式
- 返回格式

前端代码里的 fetch URL 必须和这里一致。

### 4.4 启动命令

```markdown
## 启动
- 安装：npm install / 无需安装（CDN）
- 运行：npm run dev / 直接打开 index.html
- 构建：npm run build（如需要）
```

Spec 写完不等确认，直接进入编码。

---

## 阶段 5: 写码

### 执行原则

1. **按 Spec 文件清单顺序写**，不跳不漏
2. **每写完一个文件，立即验证**：
   - 语法正确（能被解析）
   - import 路径存在
   - 如果是可构建项目，跑一次构建确认无错
3. **遇到报错立即修**，不攒到最后
4. **全部写完后执行启动命令**

### 编码约束

- 图标只用声明的图标库（Lucide / Heroicons / Tabler），禁止 emoji 作为图标
- 颜色从方案文档的视觉方向中取，不随意硬编码
- 禁止紫/粉渐变 AI 模板风格
- 前端 fetch URL 必须和 Spec 中的 API 路径完全一致
- 不写 TODO / placeholder / "后续实现" — 要么实现要么不写

### 增量验证节奏

```
写文件 A → 语法检查 ✓
写文件 B → 语法检查 ✓ → 构建检查 ✓
写文件 C → 语法检查 ✓
全部完成 → 完整构建 ✓ → 启动运行
```

---

## 阶段 6: 跑起来

代码写完后：

1. 执行 Spec 中定义的启动命令
2. 确认项目成功运行（无崩溃、无白屏、无控制台红色错误）
3. 告知用户项目已运行，给出访问方式（URL / 文件路径）
4. **等待用户反馈**

---

## 反馈迭代循环

用户反馈后：

1. 理解要改什么
2. 改代码
3. 验证改动（构建 + 运行）
4. 告知用户已改完，项目已重新运行
5. 继续等待反馈

循环直到用户满意。每轮迭代只改用户提到的部分，不主动"顺手优化"其他东西。

---

## 错误处理

### 构建/运行报错时

1. 读错误信息，定位原因
2. 如果是 import 路径错 → 检查 Spec 文件清单，修正路径
3. 如果是 API 版本不兼容 → 联网查正确用法，修正代码
4. 如果是依赖缺失 → 安装依赖，重新运行
5. 修完后重新构建/运行，确认问题解决

### 连续报错时

不要盲目重试同一个修法。停下来：
- 重新读错误信息
- 联网搜索这个具体错误
- 找到根因再改

---

## 首轮响应契约

触发 SEEAI 后第一条回复必须：

1. 说明 **"Super Dev SEEAI 赛事模式已激活"**
2. 立刻执行阶段 1（拆解需求），列出 5 点分析
3. 说明接下来会联网搜索 → 写方案文档 → 等确认 → Spec → 写码 → 跑起来
4. 如果需求足够清晰，可以在同一条回复中完成阶段 1 + 阶段 2

---

## 会话连续性

- 进入 SEEAI 后全程保持在赛事合同内
- 用户说"改/调整/换个方向"→ 留在当前阶段或回到对应阶段修改
- 用户说"重来"→ 从阶段 1 重新开始
- 只有用户明确说"退出赛事模式"或"切回正常"时才退出

---

## 与标准流程的区别

| 维度 | 标准 /super-dev | 赛事 /super-dev-seeai |
|------|----------------|----------------------|
| 文档 | PRD + 架构 + UIUX 三文档 | 一个方案文档 |
| 门禁 | 文档确认 + 预览确认 | 只有方案确认 |
| 质量闭环 | quality gate + red-team + proof-pack | 无，跑起来就行 |
| 研究深度 | 完整竞品调研 + 知识库 | 快速搜索，够用就行 |
| 编码节奏 | 前端优先 → 后端 → 测试 | 按 Spec 顺序一次性写完 |
| 迭代方式 | 阶段门禁 | 用户反馈驱动 |
