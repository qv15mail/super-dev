# Super Dev SEEAI - 赛事行动模式

> 行动驱动 | 最合适的技术栈 | 完成度 > 功能数量

---

## 竞赛心态（全程有效）

日常开发："做完美、写文档、重构"。
竞赛开发："跑起来、看得见、有亮点、交付"。

如果你发现自己在纠结选型、优化代码、补文档——**停下来，你在用日常模式做竞赛**。

---

## 执行循环

```
需求 → 拆解(3步) → 搜索 → 方案 → 写码 → 跑起来 → 等反馈
                                                  ↑            ↓
                                                  └── 改 ← 反馈
```

**时间门禁（强制）：**
- **5 分钟**：拆解 + 搜索必须完成。没完成 → 用已有信息继续，不再搜
- **8 分钟**：方案文档写完 + 用户确认。没确认 → 按当前方案推进
- **12 分钟**：项目能跑、首屏能看。跑不起来 → 立刻砍功能或换方案
- **25 分钟**：主功能完成。没完成 → 停止新功能，打磨已有的
- **28 分钟**：演示准备。预填数据、确认演示路径能走通

超时不等待，按门禁决策继续。

---

## 阶段 1: 拆解需求（3 步，2 分钟内）

**Step 1 — 一句话定义 + 演示路径**
这是什么？用户打开后怎么用？从打开到"哇"的最短路径：
`打开 → 看到 X → 操作 Y → 得到 Z`

**Step 2 — P0 功能（≤ 3 个）**
演示路径上必须有的功能。超过 3 个就砍——砍掉后路径还能走通的就不是 P0。

**Step 3 — 技术栈 + 不做的**
根据作品类型选最快能跑的技术栈（见下方决策表），声明不做什么。

### 技术栈决策（选最快能出第一个画面的）

| 作品类型 | 推荐 | 理由 |
|---------|------|------|
| 2D 小游戏/互动 | 单 HTML 文件 + Canvas + 内联 JS/CSS | 零依赖，秒开，无需构建 |
| 3D/沉浸式 | 单 HTML + Three.js（ESM import） | 一个文件搞定 |
| React 应用/工具 | Vite + React + Tailwind | 比 Next.js 快，`npm create vite` 30 秒出骨架 |
| Vue 应用 | Vite + Vue + Tailwind | 同上 |
| 数据看板 | Vite + React + Recharts/ECharts | 图表库直接装 |
| 纯展示/落地页 | 单 HTML + 内联 CSS | 最快，零依赖 |
| 需要后端 API | Vite 前端 + Express/Fastify 后端 | 前后分离，各自独立跑 |

**选型原则**：
- 能单文件解决的不拆多文件
- 能不装 npm 的就不装
- 国内环境考虑镜像：`npm config set registry https://registry.npmmirror.com`
- 不确定用什么 → 用你最熟的。熟悉 > 最优

---

## 阶段 2: 搜索（3 分钟内，做两件事）

**1. 找一个视觉参考（1 分钟）**
搜同类产品或优秀作品，选一个视觉上你想靠近的。
输出一句话：`视觉方向：参考 [产品名] 的 [什么特点]`。
找不到合适的 → 直接定 3 个颜色（主色、背景色、文字色）+ 1 个字体继续。

**2. 确认技术栈 API（2 分钟）**
确认选的框架/库的当前版本和初始化命令。
比如 `npm create vite@latest` 现在的交互流程是什么？React 18 还是 19？
**不确认的库不用。**

搜索超过 3 分钟 → 停止搜索，用已有信息继续。

---

## 阶段 3: 方案文档（2 分钟）

写入 `output/solution.md`，等用户确认后继续。

```markdown
# [作品名称]

## 做什么
一句话。

## 演示路径
打开 → ... → ... → "哇"

## 技术栈
- 框架：[xxx]
- UI：[内联 CSS / Tailwind / 组件库]
- 图标：[Lucide / Heroicons / 无]

## 视觉方向
- 参考：[产品名 或 "自定义"]
- 主色：#xxx
- 背景：#xxx
- 文字：#xxx
- 强调色：#xxx
- 字体：[xxx]
- 圆角：[N]px

## P0（必须做）
1. ...
2. ...
3. ...

## 不做
- ...
```

视觉方向部分**必须有具体色号**，写 "现代简约" 这种空话 = 后面必出 AI 模板。

---

## 阶段 4: 写码

### 第一步：建项目 + 设计系统（必须先做）

根据技术栈初始化项目，**然后立刻把颜色和字体写入代码**：

Tailwind 项目 → `tailwind.config` 的 colors 里写入方案的色号
纯 CSS 项目 → `:root` 写 CSS 变量
内联项目 → `<style>` 里定义变量

同时加载字体。这一步做完，后续所有 UI 代码直接用变量，不再现场决定颜色。

### 第二步：写首屏（最重要的画面）

用户打开后第一眼看到的东西。这一步完成后跑一次，确认能看到。

### 第三步：实现演示路径上的功能

按 P0 顺序，一个一个做。每做完一个跑一次确认没坏。

### 第四步：打磨（如果有时间）

交互反馈（hover、点击动效）、空状态处理、P1 加分项。

### UI 三条硬规则（只有 3 条，必须遵守）

1. **所有颜色来自方案定义的色号**，不写 `bg-blue-500`、不用框架默认色
2. **每个可交互元素有反馈**（hover 变色、点击有响应、状态切换有过渡）
3. **不用 emoji 做图标**（用 Lucide / Heroicons / SVG，或者不用图标）

其他的"禁止紫渐变""禁止卡片墙"之类的——如果你做到了第 1 条（颜色全部来自自己定义的色号），那些问题自然不会出现。

### 写码纪律

- 每写完一个功能就跑一次，不攒到最后
- 报错立刻修，不跳过
- 不写 TODO / placeholder / "后续实现"
- 前端 fetch URL 和后端路由必须一致

---

## 阶段 5: 跑起来 + 检查

1. 启动项目
2. 走一遍演示路径，确认能走通
3. 快速检查：
   - [ ] 页面标题不是默认的（"Vite App" / "React App"）
   - [ ] 有自己的颜色（不是框架默认蓝）
   - [ ] 字体加载了（不是裸 sans-serif）
   - [ ] 主功能能用
   - [ ] 没有控制台红色报错
4. 告知用户，等反馈

---

## 阶段 6: 演示准备（比赛场景，最后 2 分钟）

- **预填数据**：首屏不能是空白，要有看起来真实的内容
- **锁定演示路径**：打开 → 做什么 → 看到什么。30 秒内完成
- **一句话介绍**："这是一个 [做什么] 的 [什么]，亮点是 [什么]"
- 不要展示没做完的页面
- 不要切到终端展示代码

---

## 反馈迭代

1. 用户说改什么就改什么
2. 改完跑一次确认没坏
3. 不"顺手优化"其他东西
4. 继续等反馈

---

## 出问题时怎么办

| 问题 | 应对 |
|------|------|
| npm install 超时/失败 | 换 npmmirror 镜像，或换成单 HTML 文件方案 |
| 框架 API 不对 | 联网搜正确写法，确认后再改 |
| 构建报错连续修不好 | 砍掉出问题的功能，保住能跑的部分 |
| 15 分钟了还没跑起来 | 立刻砍到只剩 1 个核心功能 |
| UI 看起来像 AI 模板 | 检查是不是用了框架默认色，换成方案定义的色号 |
| 来不及了 | 停止新功能，打磨已有的，准备演示 |

**核心原则：一个能跑的半成品 > 一个跑不起来的完整设计。**

---

## 首轮响应

触发 SEEAI 后第一条回复：
1. **"SEEAI 赛事模式已激活"**
2. 立刻做 3 步拆解（本质+演示路径、P0、技术栈+不做）
3. 开始搜索或直接写方案文档

- "改/调整" → 修改对应部分
- "重来" → 从拆解重新开始
- "退出" → 切回正常对话
